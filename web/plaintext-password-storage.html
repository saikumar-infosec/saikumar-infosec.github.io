📌 Plaintext Password Storage

🧠 Description

What is the vulnerability?

Passwords are stored in the database in plaintext or using reversible encoding (Base64, weak encryption). If the database is compromised, all user credentials are immediately exposed.

Why does it occur?

Developers may skip hashing for convenience, especially during early development, and forget to implement proper storage before production.

Security principle violated: Cryptographic Failure / Data Protection

Real-world analogy: Leaving a safe unlocked with the combination written on the front – anyone who finds it gets everything.

🏷️ Classification

  • Type: Cryptographic Failure
  • OWASP: A02:2021 – Cryptographic Failures
  • CWE: CWE-256: Plaintext Storage of a Password

🎯 Attack Surface

  • ✅ Database tables (via SQL injection, backup leaks)
  • ✅ Configuration files containing passwords
  • ✅ Log files where passwords are accidentally written

🔍 Detection Methodology

  • Obtain database access via SQL injection or backup file.
  • Look at the `password` column – if it looks like readable text or short hash (MD5), it's insecure.
  • Check if password reset emails send your current password (clear sign of plaintext storage).

💣 Basic Example

SELECT username, password FROM users;
-- Returns: admin, mySecretPassword123

🚀 Advanced Exploitation

# Dump all passwords via SQL injection
' UNION SELECT username, password FROM users --

⚙️ Exploitation Steps

  1. Gain access to the database via SQL injection, exposed backup, or other means.
  2. Extract the credentials.
  3. Use them to log in as any user, including administrators.
  4. Attempt credential stuffing on other services (password reuse).

🛡️ Mitigation

✅ Hash passwords with strong algorithms: bcrypt, Argon2, PBKDF2.
✅ Use unique salts per password.
✅ Never use reversible encryption for passwords.
✅ Implement key derivation functions (KDF).
✅ Never send current password via email.

🔓 Mitigation Bypass

  • If weak hashing (MD5) is used, attackers can crack hashes offline.
  • If no salt is used, rainbow tables can crack many hashes quickly.

⚠️ Risk / Impact

Severity: Critical

Impact: Complete credential exposure, credential stuffing on other sites, identity theft.

📎 Proof of Concept (PoC)
