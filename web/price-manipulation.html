📌 Purchase Price Manipulation

🧠 Description

What is the vulnerability?

The application does not validate price parameters on the server side. Attackers modify hidden form fields, JSON payloads, or request parameters to change the price of an item (e.g., set $0.01 for a $1000 product).

Why does it occur?

Developers trust client‑side calculations, store prices in hidden inputs, or fail to re‑calculate totals on the backend.

Security principle violated: Integrity / Server‑side Trust

Real-world analogy: Changing the price tag on a product in a store – if the cashier doesn't check the actual price, you pay less.

🏷️ Classification

  • Type: Business Logic Flaw / Parameter Tampering
  • OWASP: A04:2021 – Insecure Design
  • CWE: CWE-472: External Control of Assumed‑Immutable Web Parameter

🎯 Attack Surface

  • ✅ Hidden form fields (``)
  • ✅ JSON payloads in API requests (`{"product_id":1,"price":0.01}`)
  • ✅ URL parameters (`/checkout?price=100`)
  • ✅ Cookies storing cart data

🔍 Detection Methodology

  • Intercept checkout request (Burp Suite).
  • Look for any parameter that represents price, discount, shipping, or total.
  • Modify the value to a very low amount (e.g., 0.01, 0, -100).
  • Check if the order is processed with the modified price.

💣 Basic Payloads

# Hidden form field manipulation
POST /checkout
price=0.01&product_id=123

# JSON API manipulation
{"product_id": 123, "price": 0.01, "quantity": 1}

# Negative price (may increase balance)
{"price": -1000, "product_id": 1}

🚀 Advanced Exploitation

# Manipulate discount parameters
POST /apply-coupon
{"code": "SPECIAL", "discount": 100}

# Change currency to get lower effective price
{"price": 100, "currency": "USD"} -> {"price": 1, "currency": "INR"}

⚙️ Exploitation Steps

  1. Add an item to cart and proceed to checkout.
  2. Intercept the request using Burp Suite or browser devtools.
  3. Find the price parameter and change it to a very low value.
  4. Forward the request and check if order is successful.
  5. Repeat to purchase expensive items at minimal cost (financial gain).

🛡️ Mitigation

✅ Never trust client‑side price values.
✅ Re‑calculate total on server using stored prices from database.
✅ Use signed/encrypted parameters for critical data (e.g., price).
✅ Implement server‑side validation for discounts and coupons.
✅ Log all price modification attempts.

🔓 Mitigation Bypass

  • If only client‑side validation (JavaScript) is used, attackers can bypass by disabling JS or using proxy.
  • If price is re‑calculated but discount logic is flawed, attackers can combine multiple discounts.

⚠️ Risk / Impact

Severity: High

Impact: Financial loss for business, ability to purchase expensive items for free, negative order amounts (stealing money).
