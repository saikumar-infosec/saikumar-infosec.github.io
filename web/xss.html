📌 Cross-Site Scripting (XSS)

🧠 Description

What is XSS?

Cross-Site Scripting (XSS) is a vulnerability that allows attackers to inject malicious scripts into web pages viewed by other users. When the victim's browser executes the injected script, it can steal cookies, session tokens, or perform actions on behalf of the user.

Types of XSS:

  • Reflected XSS: Payload is reflected immediately in the response
  • Stored XSS: Payload is stored on the server (database, comments, profiles)
  • DOM-based XSS: Payload executes in the DOM without touching the server
  • Blind XSS: Payload stored and executed in another context (admin panel)
  • Self XSS: Requires victim to paste the payload themselves

Security principle violated: Output Encoding / Input Validation

🏷️ Classification

  • Type: Injection
  • OWASP: A03:2021 – Injection
  • CWE: CWE-79: Improper Neutralization of Input During Web Page Generation

🎯 Attack Surface

  • ✅ URL parameters (?q=, ?search=)
  • ✅ POST body (form data, JSON)
  • ✅ Headers (User-Agent, Referer, X-Forwarded-For)
  • ✅ Cookies
  • ✅ File uploads (filename, metadata)
  • ✅ JSON/XML responses
  • ✅ JavaScript variables
  • ✅ HTML attributes

⚙️ Exploitation Steps

  1. Identify injection point (search box, URL parameter, comment form)
  2. Inject basic payload (<script>alert(1)</script>)
  3. If blocked, try alternative payloads (img, svg, body onload)
  4. Test context: HTML, attribute, JavaScript, URL
  5. Craft context-specific payload
  6. Test WAF bypasses if needed
  7. Escalate to cookie stealing or session hijacking

🛡️ Mitigation

✅ Output Encoding: Encode output based on context (HTML, attribute, JavaScript, URL)
✅ Content Security Policy (CSP): Implement strict CSP to block inline scripts
✅ Input Validation: Whitelist allowed characters
✅ HttpOnly Flag: Set HttpOnly flag on cookies
✅ Secure Flag: Set Secure flag on cookies
✅ X-XSS-Protection: Use X-XSS-Protection header
✅ Regular Security Audits: Conduct regular XSS testing
