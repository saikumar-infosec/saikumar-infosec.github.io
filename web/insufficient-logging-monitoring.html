📌 Insufficient Logging & Monitoring

🧠 Description

What is the vulnerability?

The application does not log critical events (failed logins, privilege changes, data access) or lacks real‑time monitoring. Attackers can brute force, escalate privileges, or exfiltrate data without detection.

Why does it occur?

Poor security architecture, lack of compliance requirements, or cost cutting on SIEM solutions.

Security principle violated: Detect and Respond

Real-world analogy: A bank with no security cameras – robbers can operate freely without being recorded.

🏷️ Classification

  • Type: Logging & Monitoring Failure
  • OWASP: A09:2021 – Security Logging and Monitoring Failures
  • CWE: CWE-778: Insufficient Logging

🎯 Attack Surface

  • ✅ Authentication endpoints (login, OTP, password reset)
  • ✅ Admin actions (user creation, privilege change)
  • ✅ Data export or download functions
  • ✅ API endpoints with sensitive data access

🔍 Detection Methodology

  • Attempt multiple failed logins – check if any log entry or alert appears.
  • Perform a high‑risk action (e.g., privilege escalation) and see if it's logged.
  • Check if logs are stored in an accessible location (e.g., /logs/ directory).
  • Attempt to access the application after hours – see if any monitoring alerts trigger.

💣 Basic Payloads (Testing Logging)

# Brute force login attempts
for i in {1..100}; do curl -X POST https://target.com/login -d "user=admin&pass=wrong$i"; done

# Try to access admin panel with low privileges
curl -X GET https://target.com/admin -H "Cookie: session=user_session"

🚀 Advanced Exploitation Impact

Attackers can operate for weeks or months without being noticed, leading to prolonged data theft, backdoor persistence, or ransomware deployment. Without logs, forensic investigation becomes impossible.

⚙️ Exploitation Steps (Attacker Perspective)

  1. Identify that logging is insufficient by testing small actions.
  2. Perform malicious actions (e.g., brute force, SQL injection) without fear of detection.
  3. Cover tracks by deleting or modifying logs if accessible.
  4. Maintain persistence for long‑term access.

🛡️ Mitigation

✅ Log all authentication attempts, privilege changes, and sensitive data access.
✅ Use a SIEM or centralized logging (ELK, Splunk, Sentinel).
✅ Set up alerts for suspicious patterns (multiple failures, unusual locations, high volume).
✅ Regularly review logs (daily or weekly).
✅ Protect log integrity (append‑only, tamper‑evident).
✅ Implement real‑time alerting for critical events.

🔓 Mitigation Bypass

  • If logs are stored locally, an attacker with RCE can delete them.
  • If alerts are only based on thresholds, low‑and‑slow attacks may bypass.

⚠️ Risk / Impact

Severity: Medium (but can be High if combined with other vulns)

Impact: Delayed breach detection, inability to perform forensics, prolonged attacker presence.

📎 Proof of Concept (PoC)

# Test if failed logins are logged
# After sending 10 wrong passwords, ask the admin to check logs.
# If no logs exist, vulnerability is confirmed.
