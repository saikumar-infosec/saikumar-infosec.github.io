📌 Default Credentials

🧠 Description

What is the vulnerability?

The application or underlying infrastructure (routers, databases, CMS, IoT devices) uses vendor‑default usernames and passwords (e.g., admin/admin, root/root, tomcat/tomcat). Attackers can easily guess these credentials to gain unauthorized administrative access.

Why does it occur?

Administrators forget to change default credentials during deployment, or the application is installed with hard‑coded credentials that are not enforced to be changed on first login.

Security principle violated: Weak Authentication / Insecure Defaults

Real-world analogy: Buying a safe that comes with the combination "0000" printed on the box – anyone who knows the default can open it.

🏷️ Classification

  • Type: Authentication Bypass / Weak Credentials
  • OWASP: A07:2021 – Identification and Authentication Failures
  • CWE: CWE-1392: Use of Default Credentials

🎯 Attack Surface

  • ✅ Admin panels of web applications (/admin, /wp-admin, /cpanel)
  • ✅ Database management interfaces (phpMyAdmin, Adminer)
  • ✅ IoT devices, routers, printers exposed to the internet
  • ✅ Default accounts in CMS (WordPress: admin/admin, Joomla: admin/admin)
  • ✅ Cloud service default IAM users

🔍 Detection Methodology

1. Manual Testing

  • Identify the technology (e.g., via response headers, favicon, URL patterns).
  • Look up default credentials for that technology (Google, default password lists).
  • Attempt login with common pairs: admin/admin, root/root, administrator/password, etc.

2. Automated Testing

# Hydra brute force using common default creds
hydra -L default_usernames.txt -P default_passwords.txt target.com http-post-form "/login:user=^USER^&pass=^PASS^:F=Invalid"

# Nmap http-default-accounts script
nmap --script http-default-accounts -p 80 target.com

✔ Must be reproducible

⚙️ Exploitation Steps

  1. Identify the target service/application version.
  2. Search for default credentials using online databases (https://default-password.info/ or https://cirt.net/passwords).
  3. Attempt login with discovered credentials.
  4. If successful, escalate privileges – create backdoor accounts, modify configurations, or deploy malware.

🛡️ Mitigation

✅ Force password change on first login (no default credentials).
✅ Remove or disable default accounts if not needed.
✅ Use strong, unique passwords for all administrative accounts.
✅ Implement multi‑factor authentication (MFA) for admin access.
✅ Regularly scan for default credentials using automated tools.
✅ Do not hardcode credentials in source code or configuration files.

🔓 Mitigation Bypass

  • If only the default username is changed but the password remains default, attackers can still guess the password.
  • Some applications allow bypassing password change by intercepting the request and skipping the "change password" step.

⚠️ Risk / Impact

Severity: Critical

Impact: Full administrative compromise, data breach, system takeover, lateral movement in the network.

📎 Proof of Concept (PoC)

# Using curl to test default credentials
curl -X POST https://target.com/admin/login \
  -d "username=admin&password=admin"

# If response contains "Welcome admin" or redirects to dashboard, vulnerability exists.
