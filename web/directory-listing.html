📌 Directory Listing Enabled

🧠 Description

What is the vulnerability?

Web server is configured to display directory contents when no index file (e.g., index.html) is present. This exposes the full file and folder structure of that directory, including source code, backup files, configuration files, or sensitive data.

Why does it occur?

Default server configurations (e.g., Apache Options Indexes) are not disabled, or developers intentionally leave directory listing enabled for convenience.

Security principle violated: Information Disclosure / Least Disclosure

Real-world analogy: Leaving the door to your storage room open with a full inventory list posted outside – anyone can see exactly what you have and where it is.

🏷️ Classification

  • Type: Information Disclosure
  • OWASP: A05:2021 – Security Misconfiguration
  • CWE: CWE-548: Exposure of Information Through Directory Listing

🎯 Attack Surface

  • ✅ Any directory without an index file (e.g., /uploads/, /backup/, /assets/)
  • ✅ Temporary or debug folders (/tmp/, /debug/)
  • ✅ Static asset folders with no index (/images/, /css/, /js/)
  • ✅ Third‑party plugin directories

🔍 Detection Methodology

1. Manual Browsing

  • Navigate to common directory paths: /uploads/, /backup/, /images/, /files/, /assets/.
  • If you see a list of files (instead of a 403/404), directory listing is enabled.

2. Automated Fuzzing

# Use dirb or gobuster to find directories
gobuster dir -u https://target.com -w /usr/share/wordlists/dirb/common.txt

# Check each discovered directory for listing
curl -I https://target.com/uploads/ | grep -i "200 OK"

# Look for sensitive files like .sql, .env, .git, .config, .pem, .key

⚙️ Exploitation Steps

  1. Enumerate all accessible directories using a wordlist.
  2. For each directory that returns 200 OK, check if it returns an HTML directory listing (contains "Index of /" or file list).
  3. Browse the listing and look for sensitive files: source code, database backups, configuration files, credentials, session data, logs.
  4. Download interesting files and extract further information (e.g., database credentials from config files).

🛡️ Mitigation

✅ Disable directory listing in web server configuration.
✅ Apache: Remove or comment Options Indexes; use Options -Indexes.
✅ Nginx: Use autoindex off; in location blocks.
✅ IIS: Uncheck "Directory browsing" in feature permissions.
✅ Place an empty index.html in every directory as a fallback.
✅ Use a robots.txt disallow (not a mitigation, only prevents crawling).

🔓 Mitigation Bypass

  • If directory listing is disabled but a default document (index.html) is missing, the server might still return a 403, but that does not expose the listing – so not a bypass. However, sometimes misconfigured servers still list if the request includes a trailing slash or uses specific user‑agents.

⚠️ Risk / Impact

Severity: Medium (can become High if sensitive files are exposed)

Impact: Information leakage (source code, credentials, backups), facilitating further attacks like SQL injection (if config exposed) or privilege escalation.

📎 Proof of Concept (PoC)

# Simple curl check
curl https://target.com/uploads/

# If response contains "Index of /uploads" or a list of files, vulnerability exists.
