📌 Missing Function Level Access Control

🧠 Description

What is the vulnerability?

Missing Function Level Access Control (also known as Broken Access Control on Administrative Functions) occurs when a server fails to enforce proper authorization checks for specific backend functions or API endpoints. A low‑privileged user can directly access administrative URLs (e.g., /admin/deleteUser, /api/users) and perform actions they should not be allowed to.

Why does it occur?

Developers often implement access controls only on the frontend (hiding buttons) but forget to add server‑side checks. Alternatively, they assume that a function will only be called from a legitimate page and omit authorization.

Security principle violated: Principle of Least Privilege / Defense in Depth

Real-world analogy: A bank vault that is locked, but the door to the security control room has no lock – anyone who finds the control room can open the vault.

🏷️ Classification

  • Type: Broken Access Control
  • OWASP: A01:2021 – Broken Access Control
  • CWE: CWE-285: Improper Authorization

🎯 Attack Surface

  • ✅ Administrative endpoints (/admin, /manager, /api/admin)
  • ✅ Hidden functions like /api/deleteUser, /api/updateSettings
  • ✅ Internal microservices exposed without authentication
  • ✅ Debug/developer endpoints left in production (/debug, /console)

🔍 Detection Methodology

1. Static Analysis

// Check JavaScript files for hidden admin routes
// Look for fetch('/api/admin/...') or similar

2. Dynamic Analysis

  • Enumerate common admin paths using wordlists (/admin, /api/admin/users, /manage).
  • Log in as a regular user and try to access those endpoints directly.
  • Use tools like Burp Suite Autorize or Authz to automate testing.
  • Observe response codes: 200/OK instead of 403/Forbidden indicates vulnerability.

💣 Basic Payloads

# Direct admin panel access
https://target.com/admin

# API endpoint that should be admin-only
GET /api/admin/users
GET /api/deleteUser?id=123
POST /api/updateRole {"user":"attacker","role":"admin"}

🚀 Advanced Exploitation

# Use path fuzzing to find hidden endpoints
ffuf -u https://target.com/FUZZ -w /usr/share/wordlists/dirb/common.txt

# If access control relies on a role parameter, try tampering
POST /api/admin/action
Cookie: session=user_session
Body: {"role":"admin","action":"deleteUser","id":456}

✔ Must be reproducible

⚙️ Exploitation Steps

  1. Enumerate administrative endpoints (use directory brute‑forcing or review client‑side JS).
  2. Obtain a session cookie for a low‑privileged account (register if needed).
  3. Replay requests to admin endpoints using that cookie (Burp Repeater, curl).
  4. If you receive a successful response, you have broken function‑level access control.
  5. Escalate privileges: create admin accounts, access sensitive data, or modify system settings.

🛡️ Mitigation

✅ Implement server‑side authorization checks for every function and endpoint.
✅ Use a centralized access control mechanism (e.g., role‑based middleware).
✅ Never rely on client‑side hiding of buttons or URLs.
✅ Deny by default – all endpoints should require explicit role authorization.
✅ Regularly audit API routes with automated tools (Autorize, ZAP).

🔓 Mitigation Bypass

  • If the check is only based on a request parameter (e.g., ?admin=true), an attacker can simply add it.
  • If the admin panel is on a different subdomain (e.g., admin.target.com) that shares the same authentication cookie, it may still be vulnerable.
  • HTTP method tampering: sometimes GET is protected but POST is not.

⚠️ Risk / Impact

Severity: High

Impact: Unauthorized privilege escalation, data breach (all user records), system compromise, complete account takeover.

📎 Proof of Concept (PoC)

# Request as regular user
curl -X GET https://target.com/api/admin/users -H "Cookie: session=regular_user_cookie"

# If response contains list of all users, vulnerability exists
{"users":[{"id":1,"username":"admin","role":"admin"}, ...]}
