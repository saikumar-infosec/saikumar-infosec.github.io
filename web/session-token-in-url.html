📌 Session Token Exposed in URL

🧠 Description

What is the vulnerability?

Session identifiers (cookies, JWTs) are transmitted as URL parameters instead of in secure cookies. This exposes them in browser history, referer headers, server logs, and shared links, allowing attackers to hijack sessions.

Why does it occur?

Developers often use URL parameters for simplicity, especially in single‑page applications or when implementing “remember me” functionality. They forget that URLs are stored everywhere.

Security principle violated: Secure Transmission / Session Management

Real-world analogy: Writing your ATM PIN on a sticky note attached to the card – anyone who sees the note can use the card.

🏷️ Classification

  • Type: Session Management Flaw
  • OWASP: A07:2021 – Identification and Authentication Failures
  • CWE: CWE-598: Use of GET Request Method With Sensitive Query Strings

🎯 Attack Surface

  • ✅ URLs containing `?token=`, `?session=`, `?auth=`, `?jwt=`, `?sid=`
  • ✅ Redirect URLs that leak tokens
  • ✅ Single-page applications (SPAs) storing tokens in URL fragments
  • ✅ Email confirmation links containing session tokens

🔍 Detection Methodology

1. Static Analysis

// Vulnerable code – token passed in URL
Dashboard

2. Dynamic Analysis

  • Browse the application after login and observe the URL bar.
  • Use Burp Suite to capture requests and check for tokens in `GET` parameters.
  • Check browser history for tokens.

💣 Basic Payloads (Example Vulnerable URLs)

https://example.com/dashboard?session=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
https://example.com/profile?token=abc123def456

✔ Must be reproducible

⚙️ Exploitation Steps

  1. Capture a URL containing a session token from browser history or a shared link.
  2. Copy the token and open a new private/incognito browser window.
  3. Paste the URL and access the victim's session.
  4. Perform actions as the victim without any credentials.

🛡️ Mitigation

✅ Never transmit session tokens in URLs.
✅ Use HTTP‑only, Secure, SameSite cookies for session management.
✅ Implement proper session regeneration after login.
✅ Avoid using GET for sensitive actions.
✅ Use POST or other methods for authentication and state changes.

🔓 Mitigation Bypass

  • If tokens are in URL but short-lived, an attacker must act quickly.
  • If tokens are also stored in cookies, the URL exposure might still be exploitable via referer leakage.

⚠️ Risk / Impact

Severity: High

Impact: Session hijacking, account takeover, data theft.

📎 Proof of Concept (PoC)

# Example stolen token from URL
https://example.com/dashboard?token=abc123

# Attacker uses it in another browser
curl -H "Cookie: session=abc123" https://example.com/dashboard
