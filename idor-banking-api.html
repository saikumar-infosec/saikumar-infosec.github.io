IDOR in Banking API: How I Accessed Another User's Transaction History

April 17, 2026 6 min read API Security · IDOR · Banking

Executive Summary

During a security assessment of a major bank's digital banking platform, I discovered a critical Insecure Direct Object Reference (IDOR) vulnerability in their transaction history API. This flaw allowed any authenticated user to view any other user's complete transaction history by modifying a single numeric parameter.

Vulnerability Summary:
Type: Insecure Direct Object Reference (IDOR)
Severity: Critical
Affected Endpoint: /api/v1/accounts/{accountId}/transactions

The Discovery

While testing the bank's mobile application, I analyzed API traffic through Burp Suite. The transaction history request looked like this:

GET /api/v1/accounts/184729/transactions?limit=50
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Host: api.bank.com

The parameter 184729 was the account ID. I asked: what happens if I change this to another number?

The Exploitation

I created two test accounts:

  • Account A (ID: 184729) - My primary account
  • Account B (ID: 184730) - Secondary account

Authenticated as Account A, I modified the request:

GET /api/v1/accounts/184730/transactions?limit=50
Authorization: Bearer [Token from Account A]

The server responded with Account B's complete transaction history – including transaction amounts, recipient names, timestamps, and balances.

Root Cause

The API correctly authenticated users but failed to verify ownership of the requested account:

// Vulnerable code
function getTransactions(userToken, accountId):
    userId = verifyToken(userToken)  // ✅ Valid
    return database.getTransactions(accountId)  // ❌ No ownership check
Secure Implementation:
function getTransactions(userToken, accountId):
    userId = verifyToken(userToken)
    if database.userOwnsAccount(userId, accountId):
        return database.getTransactions(accountId)
    else:
        return 403 Forbidden

Impact

  • Privacy Violation: Any user could view any other user's financial history
  • Regulatory: Violates GDPR, GLBA, and banking regulations
  • Reputational: Complete loss of customer trust

Disclosure Timeline

  • Day 1: Vulnerability discovered and reported
  • Day 2: Acknowledged as Critical severity
  • Day 5: Fix deployed to staging
  • Day 8: Fix deployed to production

Recommendations

  1. Implement proper authorization checks on every API endpoint
  2. Use UUIDs instead of sequential IDs
  3. Add rate limiting to prevent enumeration
  4. Conduct regular API security testing
