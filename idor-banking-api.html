IDOR in Banking API: How I Accessed Another User's Transaction History

April 17, 2026 8 min read IDOR · API Security · Banking

📌 IDOR in Transaction History API Endpoint

🧠 Description

What is the vulnerability?

An Insecure Direct Object Reference (IDOR) vulnerability exists in the bank's transaction history API endpoint. Any authenticated user can view any other user's complete transaction history by modifying the account ID parameter in the API request.

Why does it occur?

The API endpoint correctly authenticates users via JWT tokens but fails to verify that the authenticated user has permission to access the requested account resource. The server trusts the client-provided account ID without performing any ownership validation.

Security principle violated: Authorization

Input Validation Authorization

Real-world analogy: Imagine a hotel where guests can enter any room by changing the room number on their key card, because the front desk never checks if the guest actually booked that room.

🏷️ Classification

  • Type: Broken Access Control / IDOR
  • OWASP: A01:2021 – Broken Access Control
  • CWE: CWE-639: Authorization Bypass Through User-Controlled Key
  • CAPEC: CAPEC-566: IDOR

🎯 Attack Surface (Where to Test)

  • ✅ URL parameters (/accounts/{accountId}/transactions)
  • ✅ API endpoints (REST)
  • ❌ POST body (not applicable in this case)
  • ❌ Headers (not applicable)

🔍 Detection Methodology

1. Static Analysis (Source Code Review)

// Vulnerable code pattern identified
router.get('/api/v1/accounts/:accountId/transactions', authenticate, (req, res) => {
    const accountId = req.params.accountId;
    // ❌ Missing: check if req.user owns accountId
    db.getTransactions(accountId).then(transactions => res.json(transactions));
});

2. Dynamic Analysis (Burp Suite / Manual)

  • Intercepted request to /api/v1/accounts/184729/transactions
  • Modified account ID parameter to a different value
  • Observed successful response with another user's data

Tools used: Burp Suite Repeater

⚙️ Exploitation Steps

  1. Authenticate to the banking application as any valid user
  2. Capture the transaction history API request in Burp Suite
  3. Identify the account ID parameter in the URL path
  4. Modify the account ID to a different number
  5. Forward the request – receive another user's complete transaction history

🛡️ Mitigation

✅ Input validation: Validate account ID format and type
✅ Authorization check: Verify ownership before returning data
✅ Secure coding: Implement resource-based access control
✅ Use UUIDs: Replace sequential IDs with non-guessable identifiers 
// Secure implementation
router.get('/api/v1/accounts/:accountId/transactions', authenticate, async (req, res) => {
    const accountId = req.params.accountId;
    const userId = req.user.id;
    
    // ✅ Critical authorization check
    const ownsAccount = await db.userOwnsAccount(userId, accountId);
    if (!ownsAccount) {
        return res.status(403).json({ error: "Access denied" });
    }
    
    const transactions = await db.getTransactions(accountId);
    res.json(transactions);
});

🔓 Mitigation Bypass Attempts

  • Weak filters: No filtering was in place initially
  • Encoding tricks: Not applicable – direct parameter modification worked
  • Business logic abuse: The flaw was purely authorization-based

⚠️ Risk / Impact

Severity: Critical

Impact:
  • Data leak: Complete transaction history of any user
  • Privacy violation: Exposed financial data, recipient details, balances
  • Regulatory: Violates GDPR, GLBA, and banking regulations
  • Reputational damage: Complete loss of customer trust

📅 Disclosure Timeline

  • Day 1: Vulnerability discovered and documented
  • Day 1: Submitted detailed report via disclosure program
  • Day 2: Security team acknowledged as Critical severity
  • Day 5: Fix deployed to staging environment
  • Day 8: Fix validated and deployed to production

📋 Remediation Recommendations Provided

  • Implement proper authorization checks on every API endpoint
  • Use UUIDs instead of sequential numeric IDs
  • Add rate limiting to prevent enumeration attacks
  • Conduct regular API security testing with IDOR-specific test cases
  • Implement a bug bounty program for continuous security validation

💡 Lessons Learned

  • Authentication ≠ Authorization: Just because a user is logged in doesn't mean they should have access to everything
  • Never Trust Client Input: Account IDs provided by the client must always be validated server-side
  • API Security is Critical: Backend APIs are frequently under-tested despite exposing sensitive data
  • Defense in Depth: Even with proper authorization, additional controls like UUIDs provide valuable protection
